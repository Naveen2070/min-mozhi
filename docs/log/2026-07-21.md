## sync.\* CDC primitives — Task 4: `ast::sync_prim_lower` (expand_sync_prims)

Continuing `docs/superpowers/plans/2026-07-20-sync-cdc.local.md` on
`phase-2-correctness-consolidation-part2` (Tasks 1-3 — `Builtin` enum/parser
production, E0702/E0703 arg-shape checks, E0704/E0705 domain/placement
carve-out — already landed as `wip:` commits `ad5a0c0`/`1aa3c37`/`8ba4c01`).

Task 4 adds `crates/mimz-core/src/ast/sync_prim_lower.rs`:
`expand_sync_prims(items: &[ModuleItem]) -> Vec<ModuleItem>` rewrites every
checker-clean `sync.double_flop`/`sync.pulse` call site into ordinary
`Reg`/`On`/`Wire` items — `double_flop` splices one hidden stage reg into the
user's own `on rise(dst_clock)` block; `pulse` synthesizes a fresh toggle reg
plus its own `on rise(src_clock)` block, plus 3 sync-stage regs plus its own
`on rise(dst_clock)` block, and rewrites the wire's init to `stage1 ^ stage2`.
Registered in `ast/mod.rs` alongside `sync_loop_lower`.

**Deviation from the plan's own draft code** (flagged by the plan itself as
an open risk, not silently introduced): the plan's illustrative
implementation used a process-global `AtomicU64` counter for hidden names
(`__sync{N}_stage0`). That's non-deterministic across compiler runs (order
of calls across files, or repeated invocations in one process) and would
undermine the golden-file tests this same effort's later tasks depend on.
Went with the plan's own suggested alternative instead: hidden names are
derived from the call site's own target name (`lhs.base.name` for
`double_flop`, the wire's `name` for `pulse`) — `__sync_<target>_stage0`
etc. — which the checker already guarantees is module-unique (E0003), so
uniqueness comes for free with zero extra state, and matches
`sync_loop_lower`'s own existing precedent (namespacing off the construct's
own name, not a counter).

That naming change required its own name-collision guard: extended
`checker/names.rs`'s `collect_decls` (the same pass that already predicts
`SyncLoop`'s 4 generated names pre-lowering) to also predict
`sync.double_flop`'s 1 hidden reg name and `sync.pulse`'s 4 hidden names
directly from the raw, un-lowered `On`/`Wire` items, so a user name collision
is caught as an ordinary E0003 before `expand_sync_prims` ever runs. Two new
regression tests confirm this (`sync_double_flop_generated_name_collision_is_e0003`,
`sync_pulse_generated_name_collision_is_e0003`).

Also caught by the full-workspace gate (not a new bug, a stale doc): the
repo's own `docs_sync.rs` test flagged `docs/code/04-ast.md`'s file-layout
table as missing the new file — added a row for `sync_prim_lower.rs`.

Verification: `mimz-core` 559/559 (including the 3 lowering unit tests + 2
new collision tests), full workspace 949/949, clippy/fmt clean, prettier
clean on the touched doc.

Not done yet (Tasks 5-10 of the same plan): wiring `expand_sync_prims` into
the Verilog emitter (`emit_verilog/module.rs::flatten_items`) and the
simulator's elaborator (`mimz-sim/sim/elaborate.rs::elaborate_module`);
differential examples; spec/changelog updates marking `sync.*` shipped.

**Independent review (subagent, per explicit user direction to use the
subagent-driven approach) found a real bug before commit**:
`lower_double_flop_in_on_block` only lowered the FIRST `sync.double_flop`
call in an `on`-block via `.position()`, silently leaving any additional
call in the SAME block un-lowered — reachable, since the checker has no
"at most one per block" cap and `checker/names.rs`'s own collision
prediction already (correctly) loops over every statement. Fixed:
rewritten to collect one hidden reg per call found, looping the whole
block body. New regression test
`two_double_flop_calls_in_the_same_on_block_both_get_lowered` locks this
in. Everything else (the deterministic-naming deviation itself,
`names.rs`'s prediction correctness, idempotency, `unreachable!()` safety)
verified sound by the same review pass. Post-fix: mimz-core 560/560, full
workspace 950/950. Committed as `cfe55d0`.

## sync.\* CDC primitives — Task 5: wire `expand_sync_prims` into the Verilog emitter

`crates/mimz-core/src/emit_verilog/module.rs`'s `flatten_items` now runs
`expand_sync_prims` first, before its existing per-item loop — the same
one-line shape the plan itself specified.

**Plan gap found (subagent-driven implementation, not silently patched
over)**: the plan only anticipated `flatten_items`. It missed that
`emit_drives` (rendering combinational drives) walks the RAW,
un-flattened item list — `sync.pulse` lives in a `Wire`'s init
expression, exactly what `emit_drives` renders directly, so the raw
(unexpanded) call still reached emit-time's `unreachable!()` even after
the `flatten_items` fix. Fixed the same way the file already handles
`SyncLoop`/`ForEach` in `emit_drives`: an early check re-expands a
`sync.pulse`-initialized `Wire` on a one-item slice, recurses, `continue`s.
`sync.double_flop` needed no equivalent fix — its hidden reg/assign live
inside an `On` block, and the sequential-block renderer already reads from
the flattened list (`flat`), confirmed by both the implementer's TDD
(double_flop test passed on the `flatten_items` fix alone) and the
independent reviewer's own code trace.

Independent review (Approved with nits) verified: no duplicate
declaration/drive risk (`expand_sync_prims`/`lower_pulse` are pure
functions of the wire's own name, so two independent expansions of the
same AST node are byte-identical — confirmed by reverting the
`emit_drives` fix locally and watching the pulse test fail at the exact
`unreachable!()`, then restoring it); no infinite-recursion risk
(`lower_pulse`'s rewritten wire init is always a `Binary` XOR, never
another `Call`); real Icarus (`iverilog -g2005`) compile of both
primitives' emitted Verilog is clean with no duplicated declarations/
drives. Two non-blocking nits fixed directly: strengthened both new
tests' assertions from loose substrings to exact declaration/drive counts,
and swapped the pulse test's fixture from a domain-free `in` port (which
the real checker would reject via E0704 — `sync.pulse` requires the
signal to already be in `src_clock`'s domain, unlike `double_flop`) to a
`reg` clocked by `clk_src`, so the fixture stays representative of actual
checker-clean input even though `emit_src` itself bypasses the checker.

Verification: mimz-core 562/562, full workspace 952/952, clippy/fmt
clean.

## sync.\* CDC primitives — Task 8: Icarus differential example for `sync.pulse`

Same shape as Task 7, for the sibling primitive. Added
`examples/{english,tanglish,tamil,mixed}/sync_pulse.mimz`.

**Deviation from the task brief's literal Step 1** (flagged, not silently
patched over): the brief's exact module text feeds a bare `in src_pulse: bit`
straight into `sync.pulse(src_pulse, clk_src, clk_dst)` — this fails to
compile with E0704. `check_pulse_domain` (`crates/mimz-core/src/checker/clocks.rs`)
requires `sync.pulse`'s signal argument to be EXACTLY a register owned by
`src_clock` — unlike `double_flop`, a domain-free source is never allowed,
because the hidden toggle reg samples it synchronously in `src_clock`'s own
domain. Fixed by inserting an intermediate `reg src_reg: bit = 0` sampled by
its own `on rise(clk_src) { src_reg <- src_pulse }` block, then passing
`src_reg` (not `src_pulse`) to `sync.pulse` — the same module shape
`sync_pulse_produces_a_one_cycle_dst_pulse_after_toggle` already uses in
`crates/mimz-sim/src/sim/harness.rs`. All four flavors verified
byte-identical past the banner line (`mixed` hand-assembled by rotating
keywords across the other three, same as `mixed/sync_double_flop.mimz`).

Skipped a `tamil-pure` twin, same call Task 7 made (inventing a Tamil term
for a CDC pulse synchronizer without native-speaker review) — but Task 7's
own reasoning slipped on the supporting fact ("none of the recent examples
have one"), so this entry checked `tests/examples.rs`'s `PURE_TAMIL` array
against the actual example list before repeating the claim: 2 of the most
recently added `BASE_EXAMPLES` entries DO have a tamil-pure twin
(`tested_adder`→`tested_kuutti`, `foreach_sum`→`kootu`). The real basis for
skipping is only that it's a minority pattern (16 of 43 `BASE_EXAMPLES`),
not that recent examples never get one — noted explicitly so a future
reader doesn't inherit the wrong justification.

Registered `sync_pulse` in `tests/examples.rs`'s `BASE_EXAMPLES` (bumped
`[&str; 42]` -> `43`); regenerated `tests/golden/sync_pulse.v` with
`MIMZ_UPDATE_GOLDENS=1 cargo test --test examples goldens`, confirmed a
clean rerun without the flag. The emitted structure matches the hand-traced
lowering exactly: `src_reg` (clk_src-clocked sample), a hidden toggle reg
XORing itself with `src_reg` on every `clk_src` rise, and a 3-stage
`clk_dst` shift register (`stage0`/`stage1`/`stage2`) whose top two stages
are XORed together to re-derive a single-cycle pulse.

Same brief-mechanism correction as Task 7's Step 2: registered as a
dedicated `#[test] fn sync_pulse_matches_icarus` (Layer 2 style, not a
`differential()` call inside `our_simulator_matches_icarus_bit_for_bit` —
`SyncPulse` is also genuinely two-clock and that helper's default stimulus
only drives one). `tests/icarus/sync_pulse_tb.v` drives `clk_src` (period 4)
and `clk_dst` (period 10) as genuinely asynchronous clocks; hand-traced the
exact timing against the lowering above: `src_pulse` held high for exactly
one `clk_src` cycle samples `src_reg` at the t=26 edge; since `src_reg` and
the hidden toggle reg update in two separate `always @(posedge clk_src)`
blocks, nonblocking-assignment semantics mean the toggle only actually flips
one `clk_src` cycle later, at t=30 (reading `src_reg`'s now-updated value) —
which takes 2 `clk_dst` edges (t=35 to catch it into `stage0`, t=45 to
propagate into `stage1`) before `stage1 XOR stage2` produces a single
one-cycle-wide pulse at t=45, falling back to 0 by t=55 — verified against
real `iverilog`/`vvp` output directly (all 4 `check()` calls print `PASS`)
and confirmed the TB is not vacuously true by mutating one expectation and
seeing a real `FAIL` line. Mirrors
`sync_pulse_produces_a_one_cycle_dst_pulse_after_toggle` in
`crates/mimz-sim/src/sim/harness.rs`, but through the real compiled Verilog.

Per the brief's Step 6: `sync_double_flop_matches_icarus` (Task 7) and
`sync_pulse_matches_icarus` (this task) together satisfy the spec's §7 ask
for "at least one kernel-vs-Icarus multi-clock test that exercises an
actual crossing end-to-end" — both are real multi-clock crossings run
against real Icarus. No separate test was added beyond these two
per-primitive examples.

`REQUIRE_IVERILOG=1 cargo test --test icarus sync_pulse` passes directly.

13/13, `REQUIRE_IVERILOG=1 cargo test --workspace` 956/956, clippy/fmt
