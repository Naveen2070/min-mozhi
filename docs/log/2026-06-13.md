## Two pre-freeze syntax rulings landed (8.9 + 8.10)

The two section 8 items the v0.1.0 freeze pressured are now settled and implemented.

### Decision: Allow the safe monotonic chained comparison (8.9)

- **Context:** `a < b < c` was a hard parse error (E1109). Idea 8.9 wanted
  Python-style `0 <= x <= 100`.
- **Decision:** allow a **monotonic one-direction** chain (all `<`/`<=` or all
  `>`/`>=`); it desugars to `&&` of adjacent pairs (`0 <= x < 100` →
  `(0 <= x) && (x < 100)`), sharing the middle operand. Mixed-direction
  (`a < b > c`) and any chain with `==`/`!=` stay E1109.
- **Why:** the C trap (`(a<b)<c` silently wrong) was never legal here, so this
  is an **additive widening** — no program breaks. In hardware the shared
  operand is a combinational value, so reading it twice is identical (no
  software eval-order subtlety). Reads mathematically, on-mission for
  education. Safe-by-construction (honesty/safety preserved).
- **Impact:** `src/parser/expr.rs` (`comparison_chain`); parser tests
  swapped (`chained_comparison_is_an_error` → `monotonic…desugars` +
  `mixed_direction_chain_is_an_error` + `equality_cannot_be_chained`);
  spec/02 → v0.2.6 (section 3 + table + precedence note); test-map; `language_plan.md`
  section 9 marked done.

### Decision: Ratify `[hi:lo]` slicing as final, reject 8.10's break

- **Context:** idea 8.10 proposed replacing `data[3:0]` / `{a,b}` with
  Rust-style range slicing `data[8..16]` / spread concat — a breaking change.
- **Decision:** **keep `[hi:lo]` and `{a,b}` as the canonical, final forms**;
  do not adopt range slicing (now or via an edition).
- **Why:** `[hi:lo]` (msb:lsb) is the universal hardware convention
  (Verilog/VHDL/every textbook); the spec keeps slicing/operators "universal"
  so a student stays fluent across tools. The breaking win is aesthetic and
  **fails the "benefits the future" bar** against that cross-tool cost. A second
  range form would also clash descending-inclusive vs ascending-exclusive — a
  worse beginner trap than either alone.
- **Impact:** spec/02 → v0.2.6 (section 1.8 note); no code change (slicing already
  works); `language_plan.md` section 9 marked done. The `..` token is therefore free
  for the spread/splat family only (8.7/8.8), never ranges.

  `pothu`…) — which turned out to match the frozen `lang/keywords.toml` exactly.
- **Decision:** (1) The profile lives only on the parser and is consumed by the
  directive parser; the `File` AST has **no** profile field, so same-AST is
  structural, not asserted. (2) The same-AST test is **identical emitted
  Verilog**, not an AST dump — spans differ between twins (different source
  bytes), so a raw dump can never be byte-identical, whereas the profile-blind
  backend makes Verilog equality the exact, span-free proof. (3) Profile names
  are keywords (not contextual idents) because Tamil-script `தமிழ்` cannot be a
  Latin-only identifier (R9). (4) No `code` profile word — code-order is the
  no-directive default; inventing a Tamil "code" word would violate R9.
- **Why:** keeps the backend completely profile-blind (the spec/04 core
  promise), reuses the existing four-flavor Verilog oracle, and avoids a
  span-stripping AST-dump helper that would duplicate the AST shape.
- **Impact:** `src/parser/{mod,items}.rs`, `src/lexer/{token,keywords}.rs`,
  `lang/keywords.toml`, the VS Code grammar, spec/03 (v0.2.5) + spec/04 (v0.2),
  `docs/code/03,06,10`, plan/roadmap/README, `tests/grammar.rs` +
  `tests/fixtures/grammar/`.
