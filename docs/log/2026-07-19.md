- **Differential fuzzing v1 lands** (`tests/differential_fuzz.rs`,
  Tasks 2-3): width-tracked bottom-up expression generator emitting real
  unsigned combinational `.mimz` source text, checker-valid by
  construction — every combine step unifies operand widths via `extend()`
  before applying an operator (`+ - +% -% & | ^`, comparisons, shift,
  concat, slice-read), so no operator's own width rule needs
  special-casing in the generator. No signed/clocked/enum/bundle/fn/
  foreach in v1 — deliberately narrow, targeting exactly the class every
  real divergence bug (BUG-6, BUG-11, BUG-16) has been in. Full design:
  `docs/superpowers/specs/2026-07-18-differential-fuzzing-design.local.md`.

- **Two tests, both green at the default N=20**:
  - `differential_fuzz_generates_checker_valid_programs` — 1000 seeds,
    Icarus-independent, asserts every generated program passes
    `checker::check`. 1000/1000 valid.
  - `differential_fuzz_matches_icarus` — the real differential: 20
    generated programs (`MIMZ_DIFF_FUZZ_N` overrides the count) run
    through our own kernel (`comb::eval_outputs`) and real Icarus, values
    asserted bit-for-bit equal.

- **First real run found BUG-18** (seed 12648432, the second iteration —
  not a rare corner case): `crates/mimz-core/src/emit_verilog/expr.rs`'s
  `Builtin::Extend` arm was a pure passthrough — it rendered its argument
  unchanged and relied on Verilog's _context-determined_ implicit
  extension, which doesn't apply inside a _self-determined_ position like
  a concatenation `{...}` operand (Verilog-2005 LRM §5.4.1). A named
  signal is fine either way (it has a definite width from its own
  declaration); an unsized integer literal — including nested through
  `extend(extend(3,3),12)`, the shape the generator's own `widen()`
  produces routinely — has no self-determined width, and Icarus rejected
  it as "indefinite width". Fixed with a new local recursive resolver,
  `resolve_const_value`, that sees through nested `Builtin::Extend`/
  `Builtin::Trunc` down to a literal and renders it as an explicitly-sized
  `W'd{v}` Verilog literal at that point, instead of the bare passthrough;
  falls back to the original passthrough for genuine runtime signals
  (already safe). Task review caught a real off-by-one in the fix itself
  (`n >= 128` should have been `n >= 127` in the `Trunc` arm — the old
  guard could compute `1i128 << 127` and then subtract 1, overflowing in
  debug builds); fixed and re-reviewed clean. Filed as **BUG-18** (MEDIUM,
  FIXED) in `docs/audit/bugs.md`, same "checker accepts it, emitter/
  simulator disagree" class as BUG-6/BUG-11/BUG-16.

- **Manual confidence pass at N=500** (`MIMZ_DIFF_FUZZ_N=500`): found a
  **second**, distinct bug (seed 12648451) — `extend(p2, 15) - p1` (a
  lossless `-`, spec-correct at 16 bits per the checker/kernel's own width
  model) lands inside a concat operand; Verilog sizes a self-determined
  `-` to just the operand width (15 bits, no awareness of mimz's "grows by
  one" semantics), silently misaligning the concatenation and producing a
  wrong value (not a build error, unlike BUG-18). Masked until BUG-18 was
  fixed (the loop previously panicked at seed 12648432, never reaching
  this one) — not a regression from the BUG-18 fix. Per explicit user
  decision, filed as **BUG-19** (MEDIUM, OPEN, deferred) rather than fixed
  immediately — it doesn't block the default N=20 CI gate, and a proper
  fix likely belongs alongside BUG-17/the A1 "shared width/const-eval
  module" stage-4 item rather than as another narrow one-off patch.

- **Final state**: 886/886 tests passing (32 suites), `cargo fmt --check`

## Stage 3 (T2 v2) complete — signed-value fuzzing lands, BUG-19's class widened

- **Signed generation** (`tests/differential_fuzz.rs`): `Frag` gained a
  `signed: bool` field. Ports and literal leaves can now be `signed[N]`
  (`signed(extend(v, w))` — `extend()` of a bare literal is always
  unsigned per `call_ty`'s `CtInt` arm, so a signed leaf needs the outer
  reinterpretation cast, not a different `extend` argument). New
  `cast_to(f, want_signed)` unifies operand KIND (`signed(x)`/
  `unsigned(x)`, a no-op if already matching) before every same-width
  combine, mirroring the existing `widen()` width-unification — makes
  E0403 (signed/unsigned mixing) unreachable by construction the same way
  v1 made width mismatches unreachable. Concat casts both operands to
  unsigned first (concat never accepts `signed` directly); shift preserves
  the LHS's own kind untouched.

- **Two gaps found while landing it, both pre-existing since v1 (kind-
  independent — not signed-specific), surfaced because v2's extra RNG
  draws reshuffled which shape a given seed produces:**

  1. **`clamp()` was slicing arbitrary composite fragments.**
     `checker::check` accepts slicing ANY computed sub-expression
     (`(a + b)[5:2]`), but Verilog's part-select grammar only accepts an
     identifier before `[hi:lo]` — confirmed live, `iverilog` rejects both
     `(a & b)[2:0]` and `{a, b}[3:0]` as a syntax error. `gen_slice`
     already restricted itself to bare port identifiers; `clamp()` never
     did, and only started actually exceeding the 32-bit cap via
     `combine_concat` often enough to hit this once the operator mix
     changed. Fixed: `clamp` now only slices a genuine port identifier: a
     composite over cap is discarded and replaced with a fresh leaf.

  2. **BUG-19's "self-determined position" class is bigger than "lossless
     `+`/`-`"** — it also covers wrapping `+%`/`-%`, with a worse symptom.
     `N=500` deep-pass seed `12648524`:
     `{p0, p2} & extend((extend(3, 3) -% p2), 18)` — kernel computes `4`
     (wraps modulo 2³ at the `-%`'s own 3-bit operand width, per spec,
     THEN zero-extends). Confirmed against a standalone `iverilog` run
     that real Verilog computes `61884`: `&` is context-determined, so
     Icarus widens BOTH operands of `-%` to the 18-bit context BEFORE
     subtracting — the wraparound happens modulo 2¹⁸, not 2³, a
     completely different value, not just a lost carry bit like the
     original lossless-`+`/`-` case. Documented as an update to BUG-19
     (not filed separately — same root cause, `extend()`'s passthrough
     codegen for a non-literal argument). All four operators (`+ - +% -%`)
     now excluded from the generator's same-width-family combine step so
     this already-filed, already-deferred bug doesn't turn every default
     `cargo test` run red; `&`/`|`/`^`/comparisons stay in scope (confirmed
     immune by construction — zero/sign-extension commutes with bitwise
     ops and order comparisons regardless of when Verilog performs it).

- **Final state**: full workspace `cargo test --workspace --all-targets`
  green under `REQUIRE_IVERILOG=1`, `differential_fuzz_matches_icarus`
  clean at both N=20 (CI default) and a `MIMZ_DIFF_FUZZ_N=500` deep
  confidence pass, `cargo fmt --check` and
  `cargo clippy --workspace --all-targets -- -D warnings` both clean.

**T2 v2 milestone achieved.** No NEW signed-specific divergence turned
up — both real findings were latent, kind-independent gaps v1's own
generator already had, just reached by a different random shape. Consistent
with T1's own lesson: broadening test generation surface keeps finding
things a narrower pass missed, even when the broadening itself (signed
values) isn't where the bug turned out to live.

## Stage 3 (T2 v3) complete — clocked-design fuzzing lands, BUG-21 found and fixed

- **Clocked generation** (`tests/differential_fuzz.rs`,
  `gen_clocked_module`): `clock clk` + `reset rst`, 1-3 registers each
  driven by one `on rise(clk)` block, one combinational `out y` over
  port/register values. `reset` needs no body logic — the emitter
  auto-generates each `on` block's reset branch from every assigned
  register's own declared init value (confirmed against
  `examples/english/blinker.mimz`, which never references its own `rst`
  either). Reuses v1/v2's exact expression generator for both the
  next-state and output expressions, over a combined leaf pool
  (`ports ++ regs` — a register's current value reads exactly like an
  input port). Runs our own kernel via `elaborate_project` + `run` (the
  real engine behind `mimz sim`/`test`) against a new
  `support::clocked_testbench`, extracted from `tests/icarus.rs` (pure
  refactor — that suite's own 8 tests re-verified unaffected).

- **Three real, pre-existing (clock-independent) gaps found landing it,
  all only reachable once a register's next-state expression needed
  forcing onto its EXACT pre-declared type — something v1/v2 never had
  to do (their `out y` always took its declared type FROM whatever the
  body produced):**

  1. **`widen()` was unsound for ANY composite fragment**, not just the
     specific cases already scoped out of `SAME_WIDTH_OPS`. Seed
     `202427643`: `signed(extend({(r1[1:0] <= p0[1:0]), ...}, 2))` — the
     inner 2-bit concat's `extend()` to a wider target is an invisible
     passthrough (same BUG-19-class gap), and the outer `signed(...)`
     cast then reinterpreted the STILL-narrow bits before the register
     assignment's own extension kicked in — kernel computed `3`, Icarus
     computed `15`. Fixed: added an `atomic` field to `Frag` (true only
     for a plain identifier/slice or an already-explicit-width literal);
     `widen()` now discards a non-atomic fragment and regenerates a
     fresh exactly-sized literal instead of trusting `extend()` on it —
     the same "discard what can't be made safe" move `clamp()` already
     used for over-cap composites, generalized.
  2. **`signed(x)`/`unsigned(x)` have a self-determined argument in real
     Verilog** (confirmed against the LRM and a standalone `iverilog`
     run) — `signed(extend(x, W))` never actually widens `x` before
     reinterpreting it, atomic or not, since `extend()` contributes
     nothing syntactically for a non-literal argument. Seed
     `202427986`: `signed(extend(p1[5:3], 8))` — kernel computed `6`
     (mimz's own model: zero-extend the 3-bit slice to 8, then
     reinterpret as signed); Icarus computed `-2` reinterpreted at the
     slice's OWN 3-bit width, then sign-extended to 8 assigning into the
     reg (raw bits `254`). Fixed: `force_width` now casts FIRST, then
     widens (`widen(rng, cast_to(f, target_signed), target_w)`) —
     matching the order `combine_same_width` already happened to use
     correctly (found correct by construction there, not by testing).
  3. **BUG-21** (new): a `MIMZ_DIFF_FUZZ_CLOCKED_N=2000` extra-deep pass
     (seed `202428133`) found the simulator's `ExprKind::Slice` evaluator
     (`crates/mimz-sim/src/sim/value.rs`) propagated the SLICED BASE's
     `signed` flag to the slice result, instead of always being unsigned
     like the checker's own `slice_ty` and the sibling single-bit `Index`
     arm right above it already correctly do. `extend(p1[9:7], 9)` (p1
     `signed[10]`) sign-extended a slice that should have zero-extended.
     An isolated standalone `iverilog` repro matched the checker's OWN
     model exactly, proving the emitter was innocent this time — the
     divergence was entirely inside our own kernel. Fixed by hardcoding
     `signed: false` at both `Val`-construction sites in that arm. Filed
     as **BUG-21** (MEDIUM, FIXED) in `docs/audit/bugs.md`.

clocked_matches_icarus` clean at N=20/500/2000, the combinational
