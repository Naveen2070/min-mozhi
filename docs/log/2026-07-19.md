## Stage 3 (T2 v2) complete — signed-value fuzzing lands, BUG-19's class widened

- **Signed generation** (`tests/differential_fuzz.rs`): `Frag` gained a
  `signed: bool` field. Ports and literal leaves can now be `signed[N]`
  (`signed(extend(v, w))` — `extend()` of a bare literal is always
  unsigned per `call_ty`'s `CtInt` arm, so a signed leaf needs the outer
  reinterpretation cast, not a different `extend` argument). New
  `cast_to(f, want_signed)` unifies operand KIND (`signed(x)`/
  `unsigned(x)`, a no-op if already matching) before every same-width
  combine, mirroring the existing `widen()` width-unification — makes
  E0403 (signed/unsigned mixing) unreachable by construction the same way
  v1 made width mismatches unreachable. Concat casts both operands to
  unsigned first (concat never accepts `signed` directly); shift preserves
  the LHS's own kind untouched.

- **Two gaps found while landing it, both pre-existing since v1 (kind-
  independent — not signed-specific), surfaced because v2's extra RNG
  draws reshuffled which shape a given seed produces:**

  1. **`clamp()` was slicing arbitrary composite fragments.**
     `checker::check` accepts slicing ANY computed sub-expression
     (`(a + b)[5:2]`), but Verilog's part-select grammar only accepts an
     identifier before `[hi:lo]` — confirmed live, `iverilog` rejects both
     `(a & b)[2:0]` and `{a, b}[3:0]` as a syntax error. `gen_slice`
     already restricted itself to bare port identifiers; `clamp()` never
     did, and only started actually exceeding the 32-bit cap via
     `combine_concat` often enough to hit this once the operator mix
     changed. Fixed: `clamp` now only slices a genuine port identifier: a
     composite over cap is discarded and replaced with a fresh leaf.

  2. **BUG-19's "self-determined position" class is bigger than "lossless
     `+`/`-`"** — it also covers wrapping `+%`/`-%`, with a worse symptom.
     `N=500` deep-pass seed `12648524`:
     `{p0, p2} & extend((extend(3, 3) -% p2), 18)` — kernel computes `4`
     (wraps modulo 2³ at the `-%`'s own 3-bit operand width, per spec,
     THEN zero-extends). Confirmed against a standalone `iverilog` run
     that real Verilog computes `61884`: `&` is context-determined, so
     Icarus widens BOTH operands of `-%` to the 18-bit context BEFORE
     subtracting — the wraparound happens modulo 2¹⁸, not 2³, a
     completely different value, not just a lost carry bit like the
     original lossless-`+`/`-` case. Documented as an update to BUG-19
     (not filed separately — same root cause, `extend()`'s passthrough
     codegen for a non-literal argument). All four operators (`+ - +% -%`)
     now excluded from the generator's same-width-family combine step so
     this already-filed, already-deferred bug doesn't turn every default
     `cargo test` run red; `&`/`|`/`^`/comparisons stay in scope (confirmed
     immune by construction — zero/sign-extension commutes with bitwise
     ops and order comparisons regardless of when Verilog performs it).

- **Final state**: full workspace `cargo test --workspace --all-targets`
  green under `REQUIRE_IVERILOG=1`, `differential_fuzz_matches_icarus`
  clean at both N=20 (CI default) and a `MIMZ_DIFF_FUZZ_N=500` deep
  confidence pass, `cargo fmt --check` and
  `cargo clippy --workspace --all-targets -- -D warnings` both clean.

**T2 v2 milestone achieved.** No NEW signed-specific divergence turned

## Stage 3 (T2 v3) complete — clocked-design fuzzing lands, BUG-21 found and fixed

- **Clocked generation** (`tests/differential_fuzz.rs`,
  `gen_clocked_module`): `clock clk` + `reset rst`, 1-3 registers each
  driven by one `on rise(clk)` block, one combinational `out y` over
  port/register values. `reset` needs no body logic — the emitter
  auto-generates each `on` block's reset branch from every assigned
  register's own declared init value (confirmed against
  `examples/english/blinker.mimz`, which never references its own `rst`
  either). Reuses v1/v2's exact expression generator for both the
  next-state and output expressions, over a combined leaf pool
  (`ports ++ regs` — a register's current value reads exactly like an
  input port). Runs our own kernel via `elaborate_project` + `run` (the
  real engine behind `mimz sim`/`test`) against a new
  `support::clocked_testbench`, extracted from `tests/icarus.rs` (pure
  refactor — that suite's own 8 tests re-verified unaffected).

- **Three real, pre-existing (clock-independent) gaps found landing it,
  all only reachable once a register's next-state expression needed
  forcing onto its EXACT pre-declared type — something v1/v2 never had
  to do (their `out y` always took its declared type FROM whatever the
  body produced):**

  1. **`widen()` was unsound for ANY composite fragment**, not just the
     specific cases already scoped out of `SAME_WIDTH_OPS`. Seed
     `202427643`: `signed(extend({(r1[1:0] <= p0[1:0]), ...}, 2))` — the
     inner 2-bit concat's `extend()` to a wider target is an invisible
     passthrough (same BUG-19-class gap), and the outer `signed(...)`
     cast then reinterpreted the STILL-narrow bits before the register
     assignment's own extension kicked in — kernel computed `3`, Icarus
     computed `15`. Fixed: added an `atomic` field to `Frag` (true only
     for a plain identifier/slice or an already-explicit-width literal);
     `widen()` now discards a non-atomic fragment and regenerates a
     fresh exactly-sized literal instead of trusting `extend()` on it —
     the same "discard what can't be made safe" move `clamp()` already
     used for over-cap composites, generalized.
  2. **`signed(x)`/`unsigned(x)` have a self-determined argument in real
     Verilog** (confirmed against the LRM and a standalone `iverilog`
     run) — `signed(extend(x, W))` never actually widens `x` before
     reinterpreting it, atomic or not, since `extend()` contributes
     nothing syntactically for a non-literal argument. Seed
     `202427986`: `signed(extend(p1[5:3], 8))` — kernel computed `6`
     (mimz's own model: zero-extend the 3-bit slice to 8, then
     reinterpret as signed); Icarus computed `-2` reinterpreted at the
     slice's OWN 3-bit width, then sign-extended to 8 assigning into the
     reg (raw bits `254`). Fixed: `force_width` now casts FIRST, then
     widens (`widen(rng, cast_to(f, target_signed), target_w)`) —
     matching the order `combine_same_width` already happened to use
     correctly (found correct by construction there, not by testing).
  3. **BUG-21** (new): a `MIMZ_DIFF_FUZZ_CLOCKED_N=2000` extra-deep pass
     (seed `202428133`) found the simulator's `ExprKind::Slice` evaluator
     (`crates/mimz-sim/src/sim/value.rs`) propagated the SLICED BASE's
     `signed` flag to the slice result, instead of always being unsigned
     like the checker's own `slice_ty` and the sibling single-bit `Index`
     arm right above it already correctly do. `extend(p1[9:7], 9)` (p1
     `signed[10]`) sign-extended a slice that should have zero-extended.
     An isolated standalone `iverilog` repro matched the checker's OWN
     model exactly, proving the emitter was innocent this time — the
     divergence was entirely inside our own kernel. Fixed by hardcoding
     `signed: false` at both `Val`-construction sites in that arm. Filed
     as **BUG-21** (MEDIUM, FIXED) in `docs/audit/bugs.md`.

## Stage 3 — BUG-17 fixed: combinational slice-indexed drive

- **`record_drive`** (`crates/mimz-sim/src/sim/elaborate.rs`, the
  elaborator behind `mimz sim`/`test`) now handles `sig[hi:lo] = rhs` by
  expanding it to one `bit_drives` entry per bit position — `sig[hi:lo] =
rhs` behaves exactly like writing `sig[hi] = rhs[hi-lo]; …; sig[lo] =
rhs[0];` one bit at a time — reusing the EXISTING `bit_drives`-then-
  `Concat` assembly path unchanged. `comb.rs` (behind `mimz eval`) got the
  same treatment, plus its own `drivers` map had to change from borrowed
  `BTreeMap<String, &Expr>` to owning `BTreeMap<String, Expr>` since the
  synthesized per-bit nodes are new, not borrowed from the original AST.

- **Caught a real bug in the fix's first draft before landing it**:
  naively reconstructing each target bit as `rhs[bit_position]` breaks
  when `rhs` is compile-time-constant. `foreach_fill.mimz`'s own
  `lamps[i*8+7:i*8] = i*2`, after the `foreach` unroll substitutes a
  literal for `i`, becomes a constant-foldable RHS (e.g. `3 * 2`) — and a
  constant adapts to the slice's declared width the same way any other
  `Ty::CtInt` assignment does (the checker's `fit` check only verifies
  the folded VALUE fits, never that the expression's own "natural" width
  already equals `hi - lo + 1`). First draft panicked-clean with "bit
  index 7 is out of range for a 4-bit value" the moment it was tested
  against the real repro. Fixed by special-casing a constant-foldable
  RHS: pull each target bit from the folded `i128` value directly
  (`(v >> (b - lo)) & 1`, arithmetic shift so a negative constant
  sign-extends correctly into a wider slice) instead of indexing into the
  expression; a genuine runtime RHS still uses the original per-bit
  `Index` reconstruction (the checker already guarantees it's exactly
  the right width in that case).

- **Verified end-to-end**: `mimz sim --trace` on `foreach_fill.mimz`
  produces `lamps = 100925952`, confirmed against a standalone `iverilog`
  run of the same compiled Verilog, bit-for-bit identical. Re-enabled the
  layer-3 Icarus differential for `foreach_fill.mimz` in `tests/icarus.rs`
  (T1 had excluded it pending this fix) — green.

- **Correction, not a new finding**: BUG-17's original filing claimed
  `mimz eval` gave the same slice-specific rejection message as `mimz
sim`. Testing the actual fix showed otherwise: `mimz eval`
  (`comb.rs`) has never unrolled module-level `foreach`/`repeat` at all
  (its own header comment already says so — "deliberately a SLICE of the
  Phase 1.5 simulator...no repeat" — those items just fall through the
  item loop's wildcard arm), so a foreach-wrapped drive is invisible to
  it regardless of this fix; it now errors with the ACCURATE "signal
  `lamps` is never driven" instead. That's a separate, pre-existing,
  deliberately-scoped limitation of `mimz eval` specifically — not part
  of BUG-17 — corrected in the bug's own entry rather than left standing.

- **Final state**: full workspace `cargo test --workspace --all-targets`
  green under `REQUIRE_IVERILOG=1`, `cargo fmt --check` and `cargo
clippy --workspace --all-targets -- -D warnings` both clean.

**BUG-17 fixed.** Stage 3 remaining: the `tamil-pure/` differential

## Stage 4 — Phase A1a: shared width/kind rules landed

Root cause (per the earlier T2/BUG-17 sessions today): the checker, the
simulator, and the emitter each independently reimplement expression
width/kind semantics, and independent copies of the same rule can drift
— BUG-21 (found during T2 v3) was the exact shape: the simulator's
`ExprKind::Slice` evaluator inherited the sliced base's signedness
instead of always being unsigned, disagreeing with the checker's own
`slice_ty`. Phase A1a closes this for two operators (shift, slice) as a
scoped first step — designed via `superpowers:brainstorming`
(`docs/superpowers/specs/2026-07-19-shared-width-semantics-design.local.md`),
planned via `superpowers:writing-plans`
(`docs/superpowers/plans/2026-07-19-shared-width-rules.local.md`), and
executed via `superpowers:subagent-driven-development` (9 tasks, fresh
implementer + reviewer subagent per task, controller batching commits).

**What landed:**

- New pure module `mimz_core::width_rules` (`crates/mimz-core/src/width_rules.rs`):
  `Kind { width: u32, signed: bool }` — the minimal shape either side
  needs, independent of whether the caller has a static `Ty` (checker)
  or a concrete `Val` (simulator) — plus `shift_result`/`slice_result`,
  one function per operator rule, with their own unit tests.
- Checker integration: `checker/widths/ops.rs`'s `shift_ty` and
  `checker/widths/expr.rs`'s `slice_ty` both now call the shared
  functions for their final width/kind computation. Every existing
  diagnostic (error code, message, help text, order) preserved
  byte-for-byte — confirmed via the full existing checker test suite
  (267/267, unchanged pass count plus one new regression test, see
  below).
- Simulator integration: `mimz-sim`'s `sim/value.rs` — `binary_known`'s
  `Shl`/`Shr` arms and the `ExprKind::Slice` arm now call the same two
  functions instead of each carrying an independent (and, for slice,
  previously-buggy) copy of the rule. **BUG-21 is now structurally
  impossible to reintroduce for slice** — there is exactly one copy of
  that rule left, shared by both authorities.
- Deliberate new hardening (not a behavior change for any checker-clean
  program): the simulator now rejects a `signed` shift amount at
  runtime with an error, where before it silently ignored `r.signed`.
  No existing test exercised this path since the checker already
  rejects it — new test `shl_rejects_a_signed_shift_amount`.
- T3 conformance table (`crates/mimz-core/tests/width_rules_conformance.rs`,
  new): cross-checks the shared `shift_result` function, the checker's
  own `Ty`-level type inference (via a real parsed/checked program),
  and the simulator's own `Val`-level runtime evaluator all agree on a
  4-case shift table. Required adding `mimz-sim` as a `mimz-core`
  dev-dependency (test-only, no real circular dependency) and three
  minimal `pub` visibility bumps inside `mimz-sim` (`sim::value` module,
  `Resolver`, `eval`) to reach them from the test crate — scrutinized
  during review and confirmed not to leak any `pub(super)`-scoped
  internals (`binary_known` and friends stay crate-internal regardless
  of the module path being reachable).

**Bug found and fixed along the way** (Task 6's review, not part of the
original plan text): the checker's new `slice_ty` narrowed arbitrary-
magnitude user constant slice bounds (`h`/`l`, straight from
`const_bound`, no upper bound) to `u32` via a raw `as u32` cast _before_
the range check — unlike the declared width `n`, which the checker's
own `MAX_WIDTH` already caps far below `u32::MAX`. A slice bound like
`4294967296` (2^32) wrapped to `0` on the cast, so the checker silently
_accepted_ an invalid slice instead of raising E0406 — a real soundness
regression, present in the plan's own literal code, not an implementer
deviation. Fixed by saturating-narrowing instead (reusing the existing
`width_u32` helper from `ops.rs`, made `pub(super)`), so an out-of-range
bound saturates to `u32::MAX` — which still correctly trips
`slice_result`'s own `hi >= base_width` check — rather than wrapping
into a false accept. Regression test added
(`huge_slice_bound_that_would_wrap_u32_is_still_e0406`).

**Verification**: full workspace `cargo test --workspace --all-targets`
green (899/899) under `REQUIRE_IVERILOG=1`; deep differential-fuzz passes
clean at `MIMZ_DIFF_FUZZ_N=500` and `MIMZ_DIFF_FUZZ_CLOCKED_N=2000`
(the end-to-end kernel-vs-Icarus oracle, confirming this refactor
introduced no new divergence); `cargo clippy --workspace --all-targets
-- -D warnings` and `cargo fmt --check` both clean.

**Not done** (explicit follow-up, not oversights): Phase A1b — teaching
the emitter about Verilog's self-determined-vs-context-determined

## width_rules expansion: lossless + matched operators (Stage 4)

Continuing Stage 4's `width_rules` module (A1a landed shift/slice
earlier today) — expanded it to cover the lossless-growth family
(`+`/`-`/`*`) and the width-matching family (`+%`/`-%`/`*%`, `&`/`|`/`^`,
plus every comparison operator's operand-compatibility check), following
the same "one shared authority" pattern. Executed via
`superpowers:subagent-driven-development`, 7 tasks, plan at
`docs/superpowers/plans/2026-07-19-width-rules-lossless-matched.local.md`.

**What landed:**

- `width_rules::lossless_result(a, b, is_mul)` and
  `width_rules::matched_result(a, b)` (plus a new `RuleError::KindMismatch`
  variant) — pure functions mirroring the checker's existing
  `lossless_ty`/`matched_ty` computations.
- Checker's `lossless_ty`/`matched_ty` (`checker/widths/ops.rs`) both
  route their final width/kind computation through the shared functions.
  Every existing diagnostic (E0402/E0403, exact messages) preserved
  byte-for-byte.
- Simulator's `binary_known` (`mimz-sim`'s `value.rs`) — `Add`/`Sub`/`Mul`
  and `AddWrap`/`SubWrap`/`MulWrap`/`BitAnd`/`BitOr`/`BitXor` all now
  route through the same shared functions instead of each independently
  recomputing width/kind bookkeeping.

**BUG-22 found and fixed**: the simulator's `Sub` arm hardcoded its
result's `signed` field to `true` unconditionally — unlike its sibling
`Add`/`Mul` arms, which correctly derive it from the operands. Unsigned
`bits[N] - bits[M]` should stay unsigned per the checker's own
`lossless_ty` rule; the simulator disagreed. Routing `Sub` through
`lossless_result` fixed this as a direct consequence of sharing the rule
with `Add`/`Mul`, not a separate patch. New regression tests:
`sub_of_two_unsigned_values_is_unsigned` (confirmed failing pre-fix),
`sub_of_two_signed_values_is_signed` (a positive pin).

**A deeper finding, caught by the subagent-driven review loop itself
(Task 6)**: the original plan assumed the wrap/bitwise family's two
operands always share an identical `Kind` by the time `binary_known`
runs, mirroring the checker's `matched_ty`. A real implementer run
proved this false — a bare integer literal's simulator `Val`
(`ExprKind::Int => Val::from_int(value)`) always gets its own minimal
natural width (e.g. `1` → width 1), never pre-widened to match a wider
operand, even though the checker's static type system treats a literal
as "adapting" to the other side (a compile-time-only fiction with no
runtime counterpart). The original `wmax = l.width.max(r.width)`
computation was silently absorbing exactly this gap for all seven
width-matching arms; routing them through `matched_result`'s strict
equality check without first widening broke 18 real, currently-passing
tests (e.g. `reg r: bits[2]; r <- r +% 1`). Fixed by widening both
operands via the existing `extend_bits` helper BEFORE calling
`matched_result` (with a non-panicking `.unwrap_or(...)` fallback, not
`.expect()`, for the one remaining theoretical case — a genuine
signedness mismatch — that no known test exercises). This incidentally
also fixed a latent sign-extension bug: the pre-fix raw-bits arithmetic
never sign-extended a negative signed literal before combining it with
a wider operand, giving a wrong wrapped result for that specific shape;
the widening fix corrects this too, confirmed by hand-construction
(`reg r: signed[8] = 5; r <- r +% (-1)`: old code computed 6, wrong;
new code correctly sign-extends the literal and computes 4).
