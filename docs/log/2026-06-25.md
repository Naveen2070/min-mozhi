## LSP DX: hover, go-to-definition, completion (`phase-4-lsp-dx`)

Branch off `develop` (which carries `parse_recover` + the `Error` placeholder
nodes). Adds the editor-DX layer on top of the diagnostics-only LSP v0.
Additive and edition-safe — no grammar / keyword / spec change, independent of
the v0.1.0 release (no commit-tag/merge into it).

### Architecture — pure lib analysis + thin LSP adapter

The symbol logic is synchronous and AST-only, so it lives in the **library**
(`src/analysis.rs`): tokio-free, unit-testable without tower-lsp, reusable by the
WASM playground later. `src/lsp.rs` (binary) stays the thin tower-lsp adapter —
the same lib/bin split as today's `analyze`/`to_lsp`. All analysis works in
**byte offsets**; only `lsp.rs` converts to/from the LSP UTF-16 `Position`.

### Done (6 tasks, subagent-driven, each TDD + per-task gate)

- **`KeywordTable::canonical_spellings(flavor)`** (`src/lexer/keywords.rs`) — a
  flavor's full keyword list (one spelling per `Kw`), the completion keyword
  source.
- **Symbol index** (`src/analysis.rs`) — `build_index(&[LoadedFile]) ->
SymbolIndex`: one pass collecting a `Symbol` (name, `SymKind`, `file_idx`,
  defining span, hover `render`, enclosing `module_idx`) per definition across
  every loaded file. `Error` nodes skipped everywhere (no cascade).
- **`resolve_at`** — smallest name span covering the cursor (definition or use),
  resolved by scope: enclosing module → same module any file → file-level →
  anywhere. Cross-file, in-tree. Works on `parse_recover` partial trees.
- **Test-block resolution** (scope extension, requested mid-branch) —
  `test "…" for M { … }` blocks resolve: the module-under-test name and the
  body's driven inputs / `expect` / `tick` signals scope to M's ports, even
  cross-file (a new `same_module_any_file` tier in `resolve_at` carries this).
- **`completions`** — in-scope identifiers (enclosing module's members +
  file-level consts/enums + every module name) plus keywords in the file's
  **majority flavor** (`morph::majority_flavor` + `canonical_spellings`); no
  cross-flavor keyword leak.
- **LSP wiring** (`src/lsp.rs`) — `offset` (the inverse of `position`), a
  document-text store (`Mutex<HashMap<Url, String>>`) updated on didOpen/
  didChange/didSave, hover/definition/completion capabilities, `load_for_features`
  (parse_recover + import walk, skipping `std.*` virtual imports), and the three
  handlers mapping the lib result to `Hover` / `Location` / `CompletionItem[]`.

### Key decisions

- **Parse path split:** features run through `parse_recover` (half-typed files
  work); **diagnostics stay on the strict `parser::parse`** — no checker cascade
  on broken input, the exact UX concern the recovery branch deferred.
- **Cross-file:** resolve to definitions in the current file AND on-disk imports
  (the existing import walk). Embedded `std.*` (virtual `std:` paths) degrade
  gracefully — hover renders, go-to-def returns nothing (no real file URI).
- **Completion flavor:** majority flavor only, no cross-flavor keyword mixing
  (respects the one-flavor-per-file norm; avoids inviting the W0001 lint).

### Deferred

Dot-member completion (`inst.port`, enum variants after `.`); flavor-localized
hover render (English in v1); `ExprKind::Error` for expression-level recovery;
`did_close` cache eviction (the doc cache grows for the session).

- **Three CLI additions** (entries 4–7 of the CLI-improvements list):
  - `mimz doctor` (alias `mimz env`) — toolchain/environment health. Default targets an HDL _user_ (mimz version/edition, platform, an in-memory `compile_string` smoke test, and optional simulators iverilog/verilator/gtkwave); `--dev` adds the contributor toolchain (rustc, cargo, wasm32 target, wasm-pack, cargo-nextest, node). Exits non-zero only on a real problem (broken pipeline, unwritable temp dir, invalid `mimz.toml`) — the runtime CLI is fully in-process, so the simulators are genuinely optional and missing ones are warnings. `src/commands/doctor.rs`.
  - `mimz completions <shell>` — shell tab-completion scripts generated from the live clap tree (`crate::Cli`), so they can't drift from the real commands. New `clap_complete` dep (pure Rust, no feature gate). `src/commands/completions.rs`.
  - `mimz check --watch` — re-check on every save until Ctrl-C. New `notify` dep behind an optional `watch` feature (in `default`); like `tokio`/`tower-lsp` it must stay out of the wasm build. `src/commands/check.rs`.
  - **Per-phase timing on `--debug`** — `check`/`compile` now print `debug: timing — load …, check …, emit … (total …)`. `load` fuses lex+parse+import-resolution (the lib does them together); the finer lex-vs-parse split already exists via the criterion harness (`cargo bench`), so it was not worth instrumenting the lib.

- **Design decision — `mimz check --watch` watch scope** (record so it isn't "fixed" into a regression later):
  - Watches **directories** (entry file's dir + each transitive import's dir), non-recursively, and rechecks on any `.mimz` change in them. Directory granularity is deliberate: editor atomic-saves (write-temp-then-rename) still fire, where a single-file watch would miss them.
  - The watch set is **reconciled after every run**: new import dirs are added, and dirs that left the project are unwatched — but pruning happens **only after a successful load**. During a parse error the project's import list is unknown, so the last-good watch set is kept.
  - **Why not tighten to exact project-file matching** (to avoid a recheck when an _unrelated_ `.mimz` in a watched dir changes): it reintroduces a miss bug. If the entry file has a syntax error, the load fails and the file list collapses to just the entry; an edit to an imported file (often where the real fix belongs) would then not match and the save would be skipped — exactly the case `--watch` is most useful for. The spurious recheck the tightening would avoid is sub-millisecond and harmless. Robustness over cosmetics; left as-is on purpose. The only safe upgrade would be to retain the last _good_ file set and match against that with fallback-to-recheck — more state for a non-problem, deferred until reported.

- **`mimz init <name>`** — project scaffolding. Creates `./<name>/` with a documented `mimz.toml` and a starter `<name>.mimz`: a free-running counter module (named PascalCase from the project, `my_counter` → `MyCounter`) plus an inline `test` block that passes, so `mimz test` / `mimz compile` work immediately. Refuses to overwrite a non-empty dir; rejects names that are paths. The starter is modelled on the proven counter+test from `tests/test_run.rs`, and `tests/cli.rs` runs the generated project through `mimz test` to guard that the scaffold stays valid as the language evolves. `src/commands/init.rs`.
