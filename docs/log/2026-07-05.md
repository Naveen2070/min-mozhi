# 2026-07-05

## Feature: `loop` / `suzhal` / `சுழல்` constructs and unrolling

- **Context:** Hardware logic frequently requires bounded repetition, but writing out large `if-else` blocks is tedious and error-prone.
- **Implemented Changes:**
  - Activated the `loop` keyword (and Tanglish `suzhal` / Tamil `சுழல்`) in the keyword table and grammar.
  - Added loop unrolling in combinational `fn` bodies using continuation-threading to preserve `return`'s first-match-wins priority.
  - Added loop unrolling in clocked `on` blocks (to N copies) sharing the global `REPEAT_BUDGET`.
  - Added unrolling support to the simulator engine for both `fn` bodies (via Rust's early return) and `on` blocks (sharing `REPEAT_BUDGET`).
  - Updated the `fn_array_search` example to use `loop` instead of hand-unrolled `if` statements and validated via Icarus differential tests.
  - Strengthened loop-unroll tests in the emitter to prove nesting correctness, not just evaluation order.
  - Fixed `BitToggle` documentation example (uses `const WIDTH`, not `param` form to resolve `E0201`).

## Feature: Initial `sync loop` Grammar

- **Context:** To support higher-level state-machine generation (lightweight HLS), a `sync loop` module-item is being developed.
- **Implemented Changes:**
  - Added `ModuleItem::SyncLoop` to the AST.
  - Activated the `sync` keyword specifically for the sync-loop feature.
  - Implemented parser support for the sync loop module-item grammar.
