## Checker completion slice (Phase A of the finish-Phase-1 plan)

- **E0601/E0602 ‚Äî match exhaustiveness** (`widths.rs::check_patterns`,
  same walk as the type checks): enum matches must name every variant,
  `bit`/`bits[N]` must cover all 2^N values, or end with `_`. E0601
  names the first missing value/variant. E0602 = arm after `_`, or a
  duplicate pattern value. Anti-cascade: coverage is skipped when a
  pattern already drew E0409. Wire-`if` needed nothing ‚Äî the parser has
  always refused an expression `if` without `else`.
- **E0302 ‚Äî instantiation completeness** (`names.rs::check_inst`):
  every data input connected exactly once ‚Äî missing inputs listed in
  ONE error at the instance; duplicates flagged at the port. Clock and
  reset ports stay implicit-by-name (the emitter contract).
- **E0701 ‚Äî clock domains** (NEW pass 6, `src/checker/clocks.rs`):
  modules with <2 clocks skip instantly; otherwise regs are colored by
  their `on` block clock, wires/outs by the union of domains their
  drives reach (memoized; E0504 cycles yield the empty set). Errors:
  reading another domain inside an `on` block (direct or through any
  wire chain), and a wire whose own set mixes two domains. Module-local
  by design ‚Äî instance outputs carry no domain (deferred row).

## Decision ‚Äî exhaustiveness semantics (v0.2.3, founder-ruled)

Full enum/value coverage is exhaustive WITHOUT `_` (Rust rule). A
defensive `_` AFTER full coverage is legal and never unreachable ‚Äî it
documents bit-flip recovery; the emitted ternary chain makes the last
arm the Verilog default regardless. Only arms after `_` and duplicate
values are E0602. spec/02 bumped to v0.2.3 (section 1.3 + changelog).

## Phase B ‚Äî `repeat` elaboration + emission + ripple_adder (2026-06-12)

The last unfinished Phase 1 emitter feature. `repeat` already elaborated
in the CHECKER (widths + drivers walk every iteration), so this slice was
mostly the EMITTER, which had been erroring on `repeat` since day one.

- **E0303 (checker, `names.rs`)** ‚Äî a `repeat` body may only generate
  hardware (drives, instances, nested `repeat`s). Declaring anything
  inside it (port/`wire`/`reg`/`clock`/`reset`/`const`/`enum`/`on`) is
  E0303 with teaching help, reported at each offending item's span.
- **Emitter unrolling (`emit_verilog/`)** ‚Äî the emitter gained an
  `env: HashMap<String,i128>` (file consts ‚Üí module consts ‚Üí `repeat`
  loop vars) and **reuses `checker::consteval::eval`** (bumped to
  `pub(crate)`) rather than reimplementing const folding. An `Ident` in
  `env` renders as its literal; module **parameters stay out of `env`**,
  so they remain symbolic Verilog `parameter`s (the dumb-emitter
  invariant ‚Äî existing `counter`/`adder` output is byte-unchanged).
  Instance arrays `let fa[i] = ‚Ä¶` flatten to `fa__<i>` with outputs
  `fa__<i>_<port>`; indexed field reads `fa[i].port` and index arithmetic
  fold the same way; `REPEAT_BUDGET = 4096` guards runaway bounds.
- **Compile-time `if` folding** ‚Äî a condition that const-evaluates
  collapses to its taken branch. This is what makes the standard
  ripple-carry idiom `cin: if i == 0 { cin } else { fa[i-1].cout }`
  legal: at i==0 the dead `fa[-1]` arm is never emitted.
- **`ripple_adder` example (4 flavors)** ‚Äî a `const WIDTH = 4`
  ripple-carry adder built from `lib/full_adder` via `repeat`; exercises
  `repeat` + instance array + dotted import together. Verified
  byte-identical across english/tanglish/tamil/mixed, and **exhaustively
  under Icarus** (`ripple_adder_tb.v`, all 512 a/b/cin combinations).

## Decision ‚Äî `repeat` folds consts, not parameters; ripple_adder uses `const WIDTH` (2026-06-12)

The plan sketched `RippleAdder #(WIDTH = 4)` with `repeat i: 0..WIDTH`.
But the checker's const env (and the spec's own Chaser example, which
uses the literal `0..8` despite a `N` parameter) make clear that **module
parameters are NOT compile-time-foldable** in the dumb emitter ‚Äî they are
symbolic Verilog parameters. A `repeat` bound must fold to a concrete
count, which only literals, `const`s, and loop vars do. So the example
uses a module-level `const WIDTH: int = 4`: it folds everywhere
(ports ‚Üí `[(4)-1:0]`, bound ‚Üí `0..4`, `fa[WIDTH-1]` ‚Üí `fa__3`), the
emitted module is a concrete 4-bit datapath, and the `const` itself emits
no hardware (it is compile-time-only, spec/02 section 4). This is the
honest meaning of "compile-time unrolling, not a runtime loop": change
the const and recompile to regenerate the hardware. Parametric `repeat`
over a Verilog `parameter` (a `generate` loop) is a Phase 2 concern.
