# 2026-07-06

## Feature: `sync loop` Lowering and Validation

- **Context:** Completing the `sync loop` feature by lowering it to primitive hardware constructs and adding static checks.
- **Implemented Changes:**
  - Implemented AST lowering of `sync loop` into primitive `Port`, `Reg`, `On`, and `Drive` module items. This allows the compiler to generate the corresponding FSM, counter, accumulator, and status flags automatically.
  - Dynamically sized the generated sync loop counter using `clog2(hi)`.
  - Added shadow guarding for match-arm renaming during loop unrolling and lowering.
  - Updated the checker's name resolution to process sync loops and reused `E0003` for sync loop name collisions.
  - Documented the dual-purpose nature of the `sync` keyword: it serves both the `sync loop` feature and the planned CDC `sync.<method>()` functionality without grammar conflict (disambiguated by the following token).
  - Added width-inference support for `sync loop`: registered the 4 generated signals' types, width-checked the accumulator's init expression against its resolved type, and width-checked the loop body once (not sampled, since this body is emitted/simulated exactly once).
  - Wired `sync loop` into driver-uniqueness (its body is treated as one driver block, same as an `on` block, so if/else writes to the accumulator don't spuriously conflict) and clock-domain analysis (the accumulator is registered as driven by the loop's own clock).
  - Preserved original `sync loop` syntax across the pretty-printer, linter, keyword-transliteration, and static-analysis passes (they walk the user's own names/body, not the desugared lowering, so `mimz fmt`/keyword-reskin/analysis all see what the user actually wrote).
  - Wired `sync loop` into the Verilog emitter: `flatten_items` now desugars `SyncLoop` via `lower_sync_loop` before ports/regs/always-blocks are emitted, so a sync loop's generated interface and FSM compile to real Verilog through the existing Port/Reg/On/Drive codegen paths, with zero new codegen shapes.
  - Wired `sync loop` into the event-driven simulator: `elaborate_module` pre-lowers every top-level `SyncLoop` via `lower_sync_loop` before its worklist starts (filtering the original `SyncLoop` items out of `m.items` so they're not also processed un-lowered), so the generated regs/on-block reach `kernel.rs`'s existing `tick_edge` dispatch with zero kernel changes; added an `unreachable!` guard arm for the (now-unreachable) `SyncLoop` case so a future refactor that breaks the pre-lowering filter fails loudly instead of silently dropping the construct. The combinational-only evaluator (`comb.rs`) rejects `sync loop` cleanly, pointing users at `mimz sim`/`mimz test`. A new timing fixture test confirms a pulsed `start` produces `done` exactly `hi - lo + 1` cycles later and that a held-high `start` does not re-trigger mid-run. This was the last of the ~10 non-exhaustive `ModuleItem` match sites `SyncLoop` broke, bringing the crate to a clean `cargo build` (and a fully passing `cargo test --lib`, 477 tests) for the first time since the AST variant was introduced.

### Decision: `sync` keyword is dual-purpose (sync loop + future CDC)

- **Context:** `spec/03-keywords-trilingual.md` reserved `sync` for exactly
  one documented future purpose — CDC (clock-domain-crossing) synchronizer
  primitives, sketched as `sync.double_flop(...)` in
  `docs/Ideas/language_plan.md`. The in-progress `phase-2-sync-loop` branch
  activates that same token for an unrelated feature, the cycle-iterating
  `sync loop` FSM+counter construct — a reservation collision the sync-loop
  design doc never cross-referenced.
- **Decision:** `sync` stays a single token serving both purposes, rather
  than renaming either. The parser disambiguates by the token immediately
  following `sync` (`loop`/`suzhal`/`சுழல்` → sync-loop construct, `.` →
  future CDC call) — no grammar conflict exists.
- **Why:** four tasks of `sync loop` implementation were already committed
  under this token by the time the collision surfaced; the two constructs
  never appear in an ambiguous position relative to each other, so a rename
  would trade real rework for a hypothetical readability concern. Chose to
  document the dual purpose instead of silently leaving the reservation
  table wrong.
- **Impact:** `spec/03-keywords-trilingual.md` bumped to v0.2.22 (KW_SYNC
  added to the active table, `sync` removed from the reserved table, this
  decision recorded in that file's own changelog);
  `docs/Ideas/language_plan.md`'s CDC sketch annotated with a pointer back
  to this decision for whoever designs that construct's real grammar.
