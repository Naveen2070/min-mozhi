# Dev Log — 2026-06-24

## Decision: Native git hooks for local quality gates

- **Context:** CI-only gates meant fmt/clippy failures reached CI before the
  developer noticed. Manual `npx prettier --check` and `npx markdownlint-cli2`
  steps were easy to forget.
- **Decision:** Added native shell-based git hooks at `.git/hooks/`:
  - `pre-commit` — fast gates: `cargo fmt --all --check`, `cargo clippy --all-targets -- -D warnings`, `npx prettier --check "**/*.md"`, `npx markdownlint-cli2`.
  - `pre-push` — heavy gates: `cargo test --all-targets`, `RUSTDOCFLAGS="-D warnings" cargo doc --no-deps`.
- **Why:** Zero-dependency approach (no Node/package.json needed at the root).
  Shell scripts run in Git Bash on all platforms. Native hooks are the
  simplest thing that works.
- **Impact:** `.git/hooks/pre-commit` and `.git/hooks/pre-push` created. CI
  boundary in `.github/workflows/ci.yml` narrowed to `main` + `develop`.

## Astro v7 Migration Fix

### What was done

Upgraded `site/` from Astro v6 (v6.4.8) to Astro v7 (v7.0.2) via the official `@astrojs/upgrade` wizard.
Also bumped: `@astrojs/react` → v6.0.0, `@astrojs/vercel` → v11.0.0.

**Problem:** `npm run dev` failed immediately after the upgrade:

```
Cannot find module '@astrojs/markdown-remark' imported from 'astro.config.mjs'
```

**Root cause:** Astro v7 no longer re-exports `@astrojs/markdown-remark` transitively from
the main `astro` package. The config imports `{ unified }` from it to power the
`markdown.processor` option (required by the custom `rehypeDocLinks` plugin). The package
must now be installed explicitly.

**Fix:** `npm install @astrojs/markdown-remark` inside `site/` (resolves to v7.2.0).
Also bumped `allowScripts` entry from `esbuild@0.27.7` → `esbuild@0.28.1` to match the
new dependency tree. No changes to `astro.config.mjs` — the `processor: unified()` pattern
is the correct Astro v7 API for opting into the remark/rehype pipeline.

**Verified:** `npm run dev` starts successfully at `http://localhost:4321`.

### Decision: Retain `unified()` processor instead of migrating to Sätteri

- **Context:** Astro v7 introduces Sätteri (Rust-based renderer) as the new default Markdown
  processor, which does not support remark/rehype plugins. The site uses `rehypeDocLinks` to
  rewrite `.md` relative links to site routes.
- **Decision:** Keep `markdown.processor: unified({ rehypePlugins: [rehypeDocLinks] })`.
- **Why:** Removing `rehypeDocLinks` would break all cross-document links in published docs.
  Sätteri cannot host custom rehype transforms at this time.
- **Impact:** `site/package.json` now lists `@astrojs/markdown-remark@^7.2.0` as an explicit
  runtime dependency.

## Fix: Astro v7 compiler rejects em dashes in HTML comments

**Problem:** After loading the site, `CompilerError: Unexpected token` at
`layouts/Base.astro:56:9`. The Astro v7 Rust-based compiler (`@astrojs/compiler`) treats
the Unicode em dash character `—` (U+2014) as an unexpected token when it appears inside
an HTML comment (`<!-- ... -->`).

**Files fixed:**

- `src/layouts/Base.astro` line 55 — multi-line comment
- `src/pages/index.astro` lines 182, 185, 210, 231 — four single-line section comments

**Change:** Replaced all `—` em dashes inside HTML comments with plain `-` hyphens.
No semantic content changed; these are developer-facing code comments only.

## Release Workflow Fix: macOS Runner Upgrade

- **Problem:** The release workflow (`.github/workflows/release.yml`) for target `x86_64-apple-darwin` requested `os: macos-13`. GitHub has retired/deprecated the `macos-13` (Intel) runner, leaving the build job stuck waiting for a runner indefinitely.
- **Decision:** Updated the runner OS to `macos-latest`.
- **Why:** Apple Silicon runners (`macos-latest` / macOS 14+) support cross-compilation for `x86_64-apple-darwin` natively in Rust, and the smoke-test runs the x86_64 binary seamlessly under Rosetta 2 translation.
- **Impact:** `.github/workflows/release.yml` updated. Verified that the workflow runner configuration is correct.
