## Re-audit: Finding A — `Shl` shift-amount truncation (+ `Shr` semantic bug)

Maintainer ran a fresh security audit pass over the full simulator value path.
Found one medium defect that the initial 2026-06-14 audit had inaccurately marked
safe (hardening.md line 80, "runtime shifts […] already shift-guarded"). Re-fixed.

### Finding A — `Shl` `as u32` truncation + `Shr` semantic error

- **Where:** `src/sim/value.rs:315` — `BinOp::Shl`
- **Original code:** `l.bits.checked_shl(r.bits as u32).unwrap_or(0)`
- **Bug:** When bit ≥ 32 was set in the shift amount (`r.bits`), `as u32`
  silently truncated to the low 32 bits, producing a wrong non-zero result.
  For example, `1 << (1 << 32)` should be 0 (shift by 4_294_967_296 exceeds
  128-bit width) but `(1u128 << 32) as u32` = 0, so it became `1 << 0` = 1.
- **Sibling `Shr`:** `r.bits.min(127) as u32` avoided the truncation panic but
  had the same semantic error: a shift-by-128 became shift-by-127 instead of 0.
- **Fix:** Both now use `if r.bits >= 128 { 0 } else { … }` before the `as u32`
  cast. This is clean, correct, and makes the intent explicit.
- **Severity:** MEDIUM — reachable via `mimz sim`/`test`/WASM (skip checker).
- **Cross-codebase `as u32` scan:** All other `as u32` casts in `src/` are
  properly bounded (concat total ≤ 128, bit-index range-checked, `checked_width`
  range 1..128, `checked_index` range 0..width≤128). No other truncation bugs.
