# 2026-06-14 — Phase 1.8 grammar engine: the expression-first flips

Branch `feat/phase-1.8-grammar-flips` (off the keystone
`feat/phase-1.8-grammar-engine`). Landed three more `thamizh-order` clause
flips from `spec/04` §3 over the **same AST** as code-order: the seq
conditional (`<cond> endral { } illaiyel { }`), the if-expression
(`c endral { a } illaiyel { b }`), and match (`<expr> poruthu { }`). With the
clocked block from the keystone, four of the five §3 flips now ship; only the
**test** flip remains (deferred — see decision below).

## What changed

- **`src/parser/expr.rs`** — `expr` gains a `Profile::Thamizh` branch
  (`expr_thamizh`): parse the operand with `binary(0)`, then one-token trailing
  lookahead — `endral` → `if_expr_thamizh`, `poruthu` → `match_expr_thamizh`.
  The match arm loop is factored into `finish_match`, shared by the code-order
  and thamizh entries. `binary` is now `pub(super)` so `items.rs` can parse a
  clause head as an operand.
- **`src/parser/items.rs`** — `seq_stmt` gains a thamizh branch
  (`seq_stmt_thamizh`): parse the head, then `endral` → `seq_if_thamizh`,
  `<-` → assignment via the new `expr_to_lvalue` (reinterprets the parsed head
  as the `LValue` the code-order path gets from `lvalue()`), `=` → the existing
  E1106 teaching error. The assignment grammar is unchanged; only the parse
  route differs.
- Everything is gated on `profile == Profile::Thamizh`; the code-order path is
  byte-for-byte untouched. No new keywords, `Kw` variants, `keywords.toml`
  entries, or E-codes — `endral`/`illaiyel`/`poruthu` are the existing
  `Kw::If`/`Kw::Else`/`Kw::Match` tokens; errors reuse E1108/E1106/E1101.

## Decision: expression-first parsing, one-token lookahead after the operand

- **Context:** the keystone clocked-block flip dispatched on a _leading_ keyword
  (`Kw::Rise`). The conditional/if-expr/match flips are postpositional — the
  clause head trails the operand — so there is no leading keyword to switch on.
- **Decision:** in thamizh mode, parse the operand first with `binary(0)`, then
  decide on the single trailing token (`endral`/`poruthu`/`<-`). In a seq block
  the head may turn out to be either a condition or an assignment lvalue, so
  `expr_to_lvalue` reinterprets the parsed head as the target when `<-` follows.
- **Why:** spec/04 mandates one-token lookahead and **no backtracking**; parsing
  the operand once and classifying it afterwards honors both. Rejected: a
  speculative `lvalue()` parse with backtracking (extra parser state, forbidden
  by the spec); using full `expr()` for the head instead of `binary(0)` (would
  let a bare nested `if`/`match` act as a condition — `binary(0)` keeps the
  parenthesization rule consistent with the code-order match scrutinee).
- **Impact:** `src/parser/expr.rs`, `src/parser/items.rs`, `docs/code/03-parser.md`.

## Decision: defer the test-form flip to Phase 1.5

- **Context:** `spec/04` §3's fifth flip is the test header
  (`M() kaaga "…" sodhanai { }`) plus the flipped test-block conditional.
- **Decision:** do not implement it now; ship the other four flips.
- **Why:** the grammar engine's only oracle is byte-identical **Verilog** from a
  profile-blind backend, but `test` blocks emit no Verilog yet (the
  simulator/testbench backend is Phase 1.5, not started) and no example carries
  a `test` block — so there is no twin and no Verilog to compare. Doing it now
  would need a weaker, span-stripped AST-equality oracle. Wait until Phase 1.5
  gives test blocks a real backend.
- **Impact:** `spec/04-grammar-engine.md`, `docs/plan/phase-1.8-grammar-engine.md`.

# 2026-06-14 — Security & overflow audit

Branch `fix/security-hardening` (off the committed Phase 1.8 work). A full
read-only audit of `src/` against malicious/malformed input — overflow, panics,
unbounded recursion, resource exhaustion, memory safety — then fixed everything
real. Findings are documented in the new `docs/audit/` folder (security / bugs /
hardening, with an index README). The compiler emits hardware-level logic, so the
bar was: never crash, never corrupt memory, never silently miscompute, never
exhaust resources on a crafted file.

## What was found and fixed

- **SEC-1 (CRITICAL) — parser stack overflow.** Recursive descent had no depth
  limit; `y = (((…50k…)))` (or `!!!!…x`, or a long `else if` chain) crashed the
  process. Added a `depth` guard (`MAX_DEPTH = 64`) with `enter()`/`leave()` on
  the five recursion necks (`expr`, `unary`, `if_expr`, `seq_if`, `test_if`) and
  a new diagnostic **E1113**. (256 was tried first and still overflowed before
  tripping — each source level is ~12 Rust frames and the CLI runs on the 1 MB
  Windows main-thread stack, so the cap must suit the smallest stack.)
- **SEC-2 (HIGH) — unhardened simulator evaluator.** `sim::comb`'s `const_eval`
  was a naive copy of the checker's evaluator (raw `i128` `+ - * << >> neg`,
  truncating `as u32` casts). The `mimz eval` path skips the checker
  (`main::eval_file`), so it was the only guard — and had none → debug panic /
  release silent-wrap on a crafted const. Fixed by **delegating to the checker's
  hardened `consteval::eval`** (single source of truth), plus `checked_index`
  for bit indices/slice bounds, `checked_width` for `extend`/`trunc`,
  `wrapping_sub`/`wrapping_neg` in `binary`/`unary`, and a `u64` concat sum.
- **BUG-1 (HIGH) — already fixed in `e519690`**, recorded in `docs/audit/bugs.md`:
  a stray top-level `}` spun `file()` into an OOM hang.
- **Hardening:** `#![forbid(unsafe_code)]` (lib + bin — locks the
  no-buffer-overflow guarantee), release `overflow-checks = true`, a 32 MB
  source-size cap in `read_source`, and per-process bench temp paths.
- **Corrected the audit's own over-rated findings** after verifying against the
  code: width arithmetic (capped at `MAX_WIDTH`), `repeat` (budget 4096), import
  cycles (detected), import path traversal (not expressible) are all already
  safe. Logged in `docs/audit/hardening.md` so they are not re-investigated.

### Decision: one evaluator, not two

- **Context:** SEC-2 existed because `sim::comb` reimplemented constant
  evaluation instead of reusing the checker's, and the copy drifted (unhardened).
- **Decision:** `sim::comb::const_eval` delegates to `checker::consteval::eval`
  (already `pub(crate)`, already shared by the emitter). The naive copy is
  deleted.
- **Why:** a single hardened evaluator cannot drift out of sync again; the
  checker's version already has `checked_*` arithmetic and the E0202 overflow
  diagnostic. Rejected: re-hardening the copy in place (leaves two
  implementations to keep aligned forever).
- **Impact:** `src/sim/comb.rs`, `docs/audit/security.md`.

### Decision: conservative MAX_DEPTH + release overflow-checks

- **Context:** the depth cap must hold on the smallest stack the parser ever runs
  on (1 MB Windows main thread), and release builds previously wrapped on
  overflow silently.
- **Decision:** `MAX_DEPTH = 64` (empirically safe; 256 overflowed), and
  `overflow-checks = true` in `[profile.release]` as a defense-in-depth backstop
  behind the checked arithmetic.
- **Why:** correctness of emitted hardware is safety-critical; a loud abort beats
  a silent miscompile. A future move of parsing onto a dedicated large-stack
  thread could raise the cap for machine-generated input.
- **Impact:** `src/parser/mod.rs`, `Cargo.toml`, `docs/audit/`.

## Continuous fuzzing (follow-up, same day)

Landed the `cargo-fuzz` target the audit recommended, closing the "ongoing
assurance" gap. New standalone `fuzz/` crate (`mimz-fuzz`, own manifest +
`[workspace]` so the root `build`/`clippy`/`test` gate never sees it) with one
target `lex_parse_eval`: NFC-normalize the raw bytes (mirroring
`project::read_source`) → `lexer::lex` → `parser::parse` →
`sim::comb::eval_outputs` (empty input/param maps — still drives the SEC-2
constant-evaluation path). libFuzzer treats any panic / abort / hang as a
finding, so it asserts the audit's core guarantee by construction.

- **CI:** new `fuzz` job in `.github/workflows/ci.yml` — ubuntu + nightly,
  `cargo install cargo-fuzz`, 60 s smoke per push/PR. Bounded so it gates without
  dominating CI; a crash writes a reproducer to `fuzz/artifacts/` and fails the
  build.
- **Platform constraint:** libFuzzer needs nightly and runs on Linux/macOS only
  — it does **not** build on this Windows dev box. So the harness is **authored
  here, verified in CI** (local devs use WSL2/Linux). Verified locally only that
  the existing gate is unaffected: `fuzz/` is excluded from root cargo commands.
- Docs synced: `docs/audit/hardening.md` (Ongoing assurance → done),
  `docs/audit/README.md` (audit-log row), `docs/code/10-test-map.md` (fuzz
