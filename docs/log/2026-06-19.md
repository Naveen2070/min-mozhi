## Interactive playground waveform — stimulus controls + hover inspection

The waveform was read-only: `simulate` ran once with whatever flags were in the
command box. For a combinational adder with one fixed input that draws a flat
line — there was no way to vary inputs without hand-typing `--sweep`, and no way
to read a value off the canvas. Maintainer asked for an interactive waveform.
Decision (after a one-question scope check): **live input controls + canvas
inspection**, with per-cycle editing for _clocked_ designs deferred (it needs a
core-simulator change + an R1 spec update).

What landed:

- **`src/runner.rs` — `ports` command.** Elaborates the module and emits its
  interface as JSON: `{"module","clocked","inputs":[{name,width,signed}],
"outputs":[…]}`. The browser builds input controls from this instead of
  re-parsing source. CLI surface untouched (runner-only — it is the playground
  engine). `signal_json` renders each port; names are identifiers so no JSON
  escaping is needed.
- **`src/runner.rs` — `sim --steps "a=3,b=5;a=7,b=1"`.** Explicit per-step input
  vectors (groups split on `;`, each parsed by the shared `parse_bindings`), fed
  straight into the existing `comb_run` — distinct from `--sweep`'s cartesian
  product, so the two error if combined. Bounded by `MAX_SWEEP_VECTORS`. Rejected
  for clocked designs with a message pointing to `--in`/`--cycles`. New
  `parse_steps` helper beside `parse_sweep`.
- **`site/src/components/Playground.tsx` — stimulus panel.** On source change
  (debounced) it calls `ports`, then renders an adaptive panel: an editable
  **step table** for combinational designs (one column per step, 0/1 toggles for
  1-bit ports, value fields for buses; add/remove steps) seeded with a short
  ramp so the wave moves on load; **held-input fields + a cycles control** for
  clocked designs. Any edit re-simulates (`sim … --vcd`) and redraws live. The
  old command-box `simulate ▸` button was removed (the panel drives the wave; the
  console still runs arbitrary commands).
- **`site/src/components/WaveformViewer.tsx` — hover cursor.** Pointer-move maps
  x → time index; the canvas redraws with a vertical cursor line and a value chip
  per signal (drawn on the canvas — no DOM overlay, so no inline styles).

Verified: Rust gate green (`fmt`/`clippy -D warnings`/`cargo test` **417**, +4
runner units: `ports`×2, `sim_steps`×2), wasm32 rebuilt + web/node glue
regenerated, headless smoke test extended (asserts `ports` JSON shape and
`sim --steps`), `astro check` 0/0/0, `astro build` 23 pages. No new wasm binding —
both ride the existing `runCommand`. Nothing committed (R12).

## Step 6 groundwork: Vercel deploy CI (`.github/workflows/deploy-site.yml`)

the playground's wasm glue is generated + git-ignored and Vercel's image has no
Rust, so CI does the whole build (pinned Rust → `wasm-bindgen` `0.2.125` → Astro)
and Vercel only ingests the finished `site/.vercel/output/` via
`vercel deploy --prebuilt`.

- **Triggers:** PRs and `master` pushes touching `site/`, `crates/mimz-wasm/`,
  `src/`, or `Cargo.*` → **preview** (`*.vercel.app`, never touches prod);
  **production** only via manual `workflow_dispatch` with `target=production` —
  the public flip stays maintainer-gated (R12).
- **Actions SHA-pinned** to match `ci.yml` (`checkout`/`rust-toolchain`/
  `rust-cache`); `actions/setup-node@…48b55a # v6` (Node 22). `permissions:
contents: read`; per-ref `concurrency` with cancel-in-progress.
- **One-time setup (documented in the file header):** `cd site && vercel link`
  for the `orgId`/`projectId`, then three repo secrets — `VERCEL_TOKEN`,
  `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID`. The Vercel project's Root Directory must
  be `site`.

## Landing polish Phase 2: interactive controls on the 2D hero (no 3D)

Built the planned interactive 3D Three.js hero (procedural waveform ribbon + PCB
grid, r3f, lazy-loaded with a reduced-motion/no-WebGL fallback), then the
maintainer decided a clean 2D oscilloscope reads more professional for a language
site. Reverted the 3D scene and **removed the `three` / `@react-three/fiber` /
`@types/three` dependencies entirely** (gone from `package.json` and the lockfile).

Instead, folded the controls into the existing 2D canvas — `site/src/components/
Hero.tsx` is now a self-contained interactive oscilloscope:

- **Play/Pause**, a **Speed** slider (clock frequency), and a **Signal** switch
  (clock / counter bus / random) drive the live trace. Control values are read via
  refs inside the rAF loop so changes apply without restarting the animation.
- The grid + trace scroll by an accumulated `phase`; `counter` draws an 8-level
  staircase, `random` a deterministic per-cell hash, `clock` the square wave.
- `prefers-reduced-motion` holds a static frame and starts paused.
- Controls are real `<button>`/`<input>`/`<select>` with `aria-label`s; the canvas
  is `aria-hidden`. No inline styles — new `.hero-*` utilities in `global.css`.

`index.astro` keeps the card chrome; `<Hero>` now supplies its own stage height +
controls bar. `astro build` clean (20 pages), `astro check` 0/0/0. Uncommitted
(R12). Plan Phase 3 (alternating feature animations) still open.
