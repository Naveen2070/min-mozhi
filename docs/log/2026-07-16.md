## Structural bundle matching — final review polish (branch `phase-2-structural-interface-matching`)

Closed out four Minor findings from the whole-branch review of feature 2.9 (structural interface matching, commits `8da8de5`/`14c5e78`/`983d6e7`). None blocked merge; all optional polish.

1. **E0901 catalog row** (`docs/code/11-checker.md`): the row still described E0901 as covering "(v0.2.24) a bundle-typed `fn` call argument or `return` value has the wrong shape" — stale since feature 2.9 shipped. Investigated: E0901 is emitted exactly once, in `check_bundle_lit` (`crates/mimz-core/src/checker/widths/mod.rs`), and fires for a **bundle literal** (`{ field: expr, ... }`) missing a declared field, at every site a literal can appear (Drive RHS, `fn` call argument, `fn` return/tail — confirmed via `expect_ty`'s dedicated `BundleLit` arm and `check_return_expr`'s early branch). This is a genuinely distinct, still-live code path from E0910/E0907/E0804 (which cover passing an already bundle-typed _value_ that structurally mismatches). **Not dead code.** Reworded the row to state the literal-only scope and cross-reference E0910/E0907/E0804 for the value-passing case.

2. Added `structural_match_composes_across_fn_return_and_port_connection` (`crates/mimz-core/src/checker/tests.rs`) — threads one `HasUART`/`SensorData` pair through `check_return_ty` (fn return) then `check_inst_widths` (port connection) in one design, proving the four `bundle_shape_match` call sites compose.

3. Added `structurally_matched_fn_arg_emits_same_as_nominal_match` / `structurally_matched_fn_return_emits_same_as_nominal_match` (`crates/mimz-core/src/emit_verilog/mod.rs`), mirroring the existing Drive/port-connection emission-equality tests. **Found along the way:** bundle-typed `fn` params/returns are never flattened to per-field ports by `render_fn_decl` the way module ports/wires are (`width()` has no bundle-flatten check before the `Type::Named`/`Type::Bundle` arms) — a bare zero-param bundle name in a `fn` signature hard-errors ("unknown type ... not a declared enum"), and even the parametric form silently emits invalid Verilog (`input u;` unflattened, `pick_tx(a)` single-arg call, `as_uart(a)_tx` syntax error). Filed as a concern, not fixed (out of scope for this pass) — worked around in the new tests with a dummy bundle param to route through `Type::Bundle` instead of the hard-erroring `Type::Named` path, which still validates the invariant these tests own (emitted text doesn't vary by bundle name).

4. Added `drive_bundle_zero_required_fields_always_compatible` (`crates/mimz-core/src/checker/tests.rs`) — `bundle Empty {}` (zero required fields) is trivially satisfied by any provided bundle, confirming `bundle_shape_match`'s required-fields loop is a correct no-op at N=0.

## BUG-10 (params half) — fixed

Root-caused via `superpowers:systematic-debugging` before touching code. Confirmed via a direct CLI repro (`mimz compile`) that the function BODY already assumed flattened per-field names (`u.tx` → `u_tx` via `expr.rs`'s generic `Field` arm, unconditional on bundle-ness) — only the `input` port declaration and the call-site argument list were missing the flatten step. This meant the fix was genuinely mechanical for params, mirroring the module-header/wire bundle-flatten convention already established.

Also found, during root-cause tracing, that the RETURN half is not the same kind of bug: a Verilog `function` can only return one value — there's no Verilog syntax for a function to return multiple named outputs, so flattening a bundle return the way ports/wires/params do is not applicable at all. Fixing it for real needs inlining (a different codegen strategy), which is a feature, not a bug continuation — discussed with the user before scoping the fix down to params only, rather than silently rejecting bundle-typed `fn` returns with a new checker error (which would have made Task 4's `check_return_ty` structural-matching-for-bundle-returns arm from feature 2.9 unreachable dead code — a much bigger, unrequested change).

**Fix:**

- `render_fn_decl`'s param loop (`crates/mimz-core/src/emit_verilog/module.rs`) — bundle-typed params (`Type::Bundle` or `Type::Named` resolving to a bundle) now flatten to one `input` per field via `resolve_bundle_fields`, same convention module ports/wires use.
- `ExprKind::FnCall`'s call-site arg expansion (`crates/mimz-core/src/emit_verilog/expr.rs`) — a bundle-typed argument now expands using the **callee's** declared field names (looked up via `self.project.funcs`), not the argument's own bundle type — required so a structurally-matched but differently-named argument (feature 2.9) still resolves to the right `<arg>_<field>` wires, since flattened signal names are keyed by field name, never declaration order.
- New test `bundle_typed_fn_param_flattens_to_per_field_inputs` (`crates/mimz-core/src/emit_verilog/mod.rs`) pins both the bare and parametric bundle-param forms. Updated `structurally_matched_fn_arg_emits_same_as_nominal_match`'s stale workaround (dummy `W` param no longer needed for the arg side) and `structurally_matched_fn_return_emits_same_as_nominal_match`'s comment (clarifies params fixed, returns still open, why).
- `docs/audit/bugs.md` BUG-10 updated to PARTIALLY FIXED; `spec/02-syntax-and-grammar.md`'s Verilog-emission paragraph now states param flattening works and return flattening is an open, tracked gap (not silently broken); `docs/plan/phase-2-ir-synthesis.md` backlog gained a new line item for the inlining feature.
