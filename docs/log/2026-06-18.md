## Plan: pull the Phase 4 web playground forward to lead the v0.1.0 public launch

Founder asked whether to build a landing page + docs — and, if possible, a
playground to run mimz and see the waveform — before going public. Verified
feasibility against the code and locked a plan.

### Decisions (founder)

- **Build one static site — landing + docs + in-browser playground — _before_ the
  public flip** (launch with the differentiator, not a README alone). Founder-gated
  (R12); the public flip + `v0.1.0` tag remain Workstream D.
- **Custom Astro app** (not Starlight): content-collections render the existing
  `docs/guide/*` + `docs/spec/*` markdown (sourced, never duplicated) and islands
  host the playground.
- **Monorepo, not a separate web repo** — docs stay a single source of truth, the
  WASM playground can't version-drift from the compiler (built against the same
  `src/lib.rs`, byte-equal differential test in one suite), versioning is atomic. The
  Node toolchain is isolated under `site/`; `crates/mimz-wasm` stays **out of
  `default-members`** so the R8 gate never builds it for the host.
- **Waveform: a custom canvas/SVG renderer behind a swappable `WaveformViewer({vcd})`
  boundary**, with **Surfer** (surfer.dev) as the documented drop-in upgrade if VCD
  later gains x/z or large traces. The VCD string is the stable contract.
- **Deploy: Vercel on a founder-owned subdomain** (not GitHub Pages) — served at root
  so no `base`-path WASM/asset pain, first-class Astro, per-branch preview deploys for
  founder review. Build wrinkle: Vercel's image lacks Rust+`wasm-pack`; leaning toward
  building wasm+site in the SHA-pinned GitHub Actions and `vercel deploy --prebuilt`
  (reproducible, pinned toolchain), with Vercel-native rust-bootstrap as the simpler
  fallback. Decided at Step 6. **Note:** any deploy (even a preview) makes the _site_
  public while the _repo_ stays private — the "go public" gate still governs the repo
  flip + `v0.1.0` tag (R12).
- **Landing may be a little flashy** — a domain-themed animated hero (gates/waveforms),
  but as a lazy-loaded island with a `prefers-reduced-motion` static fallback and a
  light WebGL/SVG approach over heavy Three.js (audience: college labs). Deferred to
  Step 6 polish so it can't delay the functional site.

### Why it's low-risk

Exploration confirmed the compiler+simulator lib is **WASM-ready today** — pure-sync,
no C deps, no `build.rs`, nothing that breaks `wasm32`. The whole pipeline is already
string-in / string-out and in-memory (`lex → parse → check → transliterate →
Project::from_files → emit`; `elaborate_project → run/comb_run → to_vcd`). The only
net-new library code is a ~30-line `compile_string(source, imports)` wrapper that
skips the `std::fs` file-loading path. The WASM browser playground is already the #1
priority in [`phase-4-ecosystem.md`](../plan/phase-4-ecosystem.md), and its
prerequisites (lib/bin split, `--json` diagnostics) landed in Phase 1.

### Site: link rewriter now routes non-`.md` repo links to GitHub (+ Windows path fix)

The docs link to repo files the site doesn't publish (`keywords.toml`, `demo/`,
`editors/`, `docs/*.md`, etc.) and to directories (`../../spec/`). The rewriter
previously handled only `.md` links, so those rendered as dead relative paths.
Extended `site/src/lib/rehype-doc-links.mjs`:

- non-`.md` repo links → **GitHub** — files to `blob/master/<path>`, directories
  to `tree/master/<path>` (trailing slash or extension-less basename ⇒ directory);
- published **section directories** (`spec/`, `docs/guide/`) → their on-site
  landing routes (`/spec`, `/guide`);
- external / `mailto:` / in-page-anchor / site-absolute links left untouched;
  links that escape the repo are left as-is.

**Bug found and fixed:** the resolver used `path.posix.resolve()`, which treats a
Windows `D:/…` path as relative and falls back to the build cwd — so on Windows the
within-repo checks silently failed (only same-dir/basename cases happened to work).
Rewrote it to resolve to repo-relative form via `path.posix.join(relDir, rel)`
(never `resolve`), keyed off the repo root derived from the `/docs/` or `/spec/`
segment. **Caching note:** Astro's content layer caches rendered HTML by source
hash and does **not** invalidate on markdown-plugin changes — a full rebuild
required clearing `site/.astro` **and** `site/node_modules/.astro`.

Verified in `dist`: zero remaining relative `../` links; `../../keywords.toml` →
`blob/master/keywords.toml`, `../../demo/` → `tree/master/demo`,

### Website Phase 2 — WASM groundwork (`compile_string` + `crates/mimz-wasm`)

Build steps 1–2 of `docs/plan/phase-4-web-presence.md`: give the browser
playground a filesystem-free compile path and a wasm entry point.

- **`mimz::compile_string(&str) -> Result<String, String>`** (`src/lib.rs`) — the
  in-memory pipeline (NFC → lex → parse → check → transliterate → emit), the same
  stages as `mimz compile` minus file I/O. Single-file: `import` is rejected with a
  plain message (no filesystem to resolve against). Errors come back as rendered,
  caret-annotated diagnostics (English). A free function, not a module, so the
  lib-module/docs-table sync (`tests/docs_sync.rs`) is unaffected.
- **`crates/mimz-wasm`** (new) — wasm-bindgen `compileToVerilog(source)`: returns
  Verilog or throws a JS `Error` carrying the diagnostics. `+ README.md`,
  `smoke-test.cjs` (headless Node proof), `test.html` (browser demo).
- **Workspace + feature-gating** — root `Cargo.toml` gains `[workspace]` (member
  `crates/mimz-wasm`). The CLI-only deps that don't build on wasm32 — `tokio`,
  `tower-lsp`, `memory-stats` — are now **optional** behind features
  (`default = ["lsp", "bench"]`); `main.rs` `#[cfg(feature = "lsp")]`-gates the LSP
  subcommand + `mod lsp`; the `mimz-bench` bin declares `required-features =
["bench"]`. The wasm crate depends on `mimz` with `default-features = false`, so
  no tokio reaches wasm32.
- **+5 integration tests** (`tests/compile_string.rs`); suite 404 → 409 (test-map +
  README badge + how-the-compiler-works updated).

Verified: full native gate — `cargo fmt`, `cargo build`, `cargo build
