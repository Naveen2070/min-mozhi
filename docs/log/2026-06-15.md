# 2026-06-15 — Robustness follow-up to the 2026-06-14 batch audit

A three-script audit (plus manual code verification) of the 2026-06-14 batch
(built-ins, `fmt`, `--lang`/`morph`, LSP localization, `translate --order`,
security hardening, fuzz) found **no correctness bugs** and confirmed the
security pass (SEC-1..4) is sound and regression-tested. This entry records the
follow-up that closed the one real robustness issue, two low-severity edge
cases, and the cheap missing-test / doc-drift items the audit surfaced.

## Production fixes

- **Atomic `mimz fmt` write** (`src/main.rs` `fmt_file`). `fmt` overwrote its
  input in place with `std::fs::write`, so an interrupted / failing write could
  truncate the user's source. Now writes a sibling `<out>.<pid>.tmp` then
  `std::fs::rename`s over the target (cross-platform atomic replace; the PID
  suffix avoids concurrent-run collisions). The write still happens only after a
  successful lex + translate, so malformed input never reaches it.
- **`morph::inflect` empty-stem guard** (`src/morph.rs`). An empty `name` (e.g.
  a diagnostic whose span did not resolve, via `localized_msg`'s
  `src.get(..).unwrap_or("")`) produced a suffix-only fragment like `ஐ …`. Now
  returns empty for an empty stem.

## Mixed-flavor warning (W0001) — the first non-fatal lint

After a feasibility discussion on SystemVerilog/CPU support and Tamil-script
identifiers (findings: synthesizable HDL is the same Verilog pipeline industry
uses — Min-Mozhi already emits Verilog-2005, so the gap is maturity not
architecture; OOP/classes are verification-only and non-synthesizable; Tamil
identifiers already compile via the emit-time transliterator), it was requested
for a rule reflecting that **English and Tanglish share code order (SVO) and may
mix, but Tamil reads differently** — so mixing Tamil keywords with the others
should be flagged. Implemented as a **non-fatal warning**, surfaced in the CLI
and the LSP.

- **`Diag` gained a `Severity`** (`Error`/`Warning`, `src/diag.rs`). `Diag::new`
  defaults to `Error` (every existing call site unchanged); `as_warning()` flips
  it. The renderer prints `error[..]` / `warning[..]` per severity; `JsonDiag`
  gained an additive `"severity"` field. The decision was a real fork — the diag
  system was errors-only and `fmt --strict` warned via a raw `eprintln!` the LSP
  can't see, so a true `Diag`-level warning was needed to reach editors.
- **`morph::flavor_mix_warning(tokens)`** — fires iff Tamil is present **and**
  `flavors_used` has >1 flavor (en+tanglish alone stays clean). Span points at
  the first keyword of the minority flavor. Code **W0001**; reuses
  `morph::flavors_used`/`majority_flavor`.
- **Non-fatal wiring** (`src/main.rs`): `check`/`compile`/`eval` compute the
  warning after lexing, render it alongside any errors, and exit FAILURE **only
  if some diagnostic is an error** (`Output::exit_for`) — a warning-only run
  succeeds and still writes Verilog. `examples/mixed/*` now emit W0001 on every
  compile (they deliberately mix all three) — exit stays 0, Verilog unchanged,
  the byte-identity test still passes.
- **LSP** (`src/lsp.rs`): `to_lsp` maps `Severity::Warning →
DiagnosticSeverity::WARNING`; `recheck` appends the per-file mix warning.
- **W-codes** are deliberately outside `ALL_CHECKER_CODES` (no forced
  error-fixture; the catalog regex matches `E`-rows only).

Tests (+6, 259 → **265**): 2 `morph` lib unit (the fire/clean matrix; W0001 is

warns + exits 0; single-flavor is clean; `--json` carries `"severity":

### Decision: fully-Tamil examples as an additive showcase, not a folder migration

- **Context:** the goal was to have examples written entirely in Tamil (keywords AND
  identifiers). The tempting route — making each flavor folder language-pure —
  was investigated.
- **Decision:** keep the four-flavor byte-identical corpus untouched; add a
  separate `examples/tamil-pure/` showcase validated by equivalence-to-counterpart
  - its own goldens/testbenches. Add identifier romanization to `translate` as an
    **opt-in** flag only.
- **Why:** language-pure folders are a **breaking change** — localized
  identifiers transliterate to different Verilog, so they cannot be byte-identical
  (violates **R9**), and they break the `mimz translate` oracle in
  `tests/translate.rs` (which asserts `translate(english/X)` matches committed
  `tamil/X` token-for-token, and `TokKind::Ident` carries the name). The rule is
  also referenced across ~46 files incl. `spec/03`, `spec/04`, architecture. The
  additive route delivers fully-Tamil code with **zero** breakage. Opt-in
  romanization keeps the lossless round-trip the default, since transliteration is
  one-way. Rejected: full pure-folder migration (too large, breaks R9 + the
  translate oracle); automatic romanization (would make the round-trip lossy by
  default).
- **Impact:** new `examples/tamil-pure/` + goldens + testbenches; `tests/`
  (`examples.rs` +2, `icarus.rs` +1 via a shared `run_self_checking` helper,
  `translate.rs` +3 integration, `src/translate.rs` +2 unit); `src/translate.rs`
  (`translate_opts`/`TranslateOpts`/`build_rename_map`); `src/emit_verilog`
  (`romanize` exposed); `src/main.rs` (`--romanize-names`). Docs: **R9** gained a
  language-pure-showcase exception (`docs/RULES.md`); `spec/04` section 2 + this
  log document the flag; `docs/code/13-tooling.md`, `examples/README.md`, and
  `docs/code/10-test-map.md` updated.

### Decision (planned, not built): per-file sidecar name-map for lossless romanized round-trips

- **Context:** `--romanize-names` is one-way, so `tamil → tanglish → tamil` with

## Sidecar name-map — romanization is now reversible

Built the deferred name-map (the decision above is now done). The one-way
`--romanize-names` becomes a lossless round-trip with a per-file sidecar.

- **`src/translate.rs`:** factored the span-walk into a private `reskin(src,
tokens, target, sub)` (identifier substitution closure), backing three paths.
  New `pub fn romanize_with_map(src, target) -> (String, NameMap)` (forward +
  inverted Latin→Tamil map) and `pub fn restore_with_map(src, target, &NameMap)`
  (reverse). `NameMap { version, names: BTreeMap }` is serde-serializable
  (deterministic, sorted JSON). `translate_opts(romanize_names)` now delegates to
  `romanize_with_map`.
- **`src/main.rs`:** `mimz translate` gained `--names-map <FILE>`. Forward with
  `-o <out>` writes `<out>.names.json` (only when there are Tamil names);
  stdout-only prints a note that no map is written. Reverse loads the JSON
  (`load_name_map`, clean error on missing/malformed) and restores. `--names-map`
  - `--romanize-names` is a clean error (opposite directions); both warn-and-skip
    under `--order`. Sidecar path appends `.names.json` (never replaces the
    extension).
- **Round-trip:** `Tamil → (romanize, capture map) Latin → (restore via map)
Tamil` is byte-for-byte the canonical Tamil source.

## `mimz.toml` config + name-map auto-discovery — stop retyping flags

Two convenience features so per-project flags aren't repeated every invocation.

- **Auto name-map discovery** (`src/main.rs` `translate_file`). A reverse reskin
  now auto-loads the `<input>.names.json` sidecar when it sits next to the file —
  `mimz translate k.mimz --to tamil` restores Tamil names with no `--names-map`.
  Explicit `--names-map <path>` overrides; `--no-names-map` opts out; `--order`
  and `--romanize-names` runs never auto-restore. A `note:` prints on auto-load.
- **`src/config.rs` (new `config` module)** — per-project defaults from
  `mimz.toml`. `Config { lang, [translate], [fmt] }`, all fields `Option`,
  `deny_unknown_fields` (typos error). `discover()` walks up from the input file
  (canonicalized) to the nearest `mimz.toml`; the global `--config` overrides.
  Missing file ⇒ defaults; malformed ⇒ clean error (per-project, so not a panic
  like the embedded tables). TOML to match `lang/keywords.toml`; the name-map sidecar
  stays JSON (machine-written).
- **CLI merge** (`src/main.rs`). Every subcommand resolves the config and merges
  **CLI flag › config value › built-in default**: `lang` (check/compile/eval),
  `[translate]` to/order/romanize_names/names_map, `[fmt]` to/strict. New global
  `--config`, new `--no-names-map`.

### Decision: config precedence and discovery

- **Context:** users were repeating `--to`, `--lang`, `--names-map` every run.
- **Decision:** a walk-up-discovered `mimz.toml` (TOML, `--config` overrides),
  precedence **CLI › config › default**; name-map auto-discovery on by default
  for reverse translate.
- **Why:** matches Cargo/rustfmt ergonomics; TOML matches the project's other
  human-authored tables; CLI-wins keeps one-off overrides trivial. Rejected: a
  central name-map (cross-file collisions) and JSON config (no comments).
- **Impact:** new `src/config.rs` + `pub mod config` (lib.rs crate-map + docs/code
  README row, `tests/docs_sync.rs`); `src/main.rs` global `--config`,
  `--no-names-map`, per-command merge; docs/code/13-tooling.md (`config` section +
  auto-discovery), examples/README.md, this log, 10-test-map.md.

## Fuzz / security audit — changes since 2026-06-14

- **Scope:** the since-yesterday diff (`config`, reversible romanization +
  sidecar name-map, W0001 lint + error-language `morph`). Goal: overflow +
  critical security. Windows has no libFuzzer (nightly + Linux/macOS only), so a
  deterministic LCG stress harness drove the new `translate`/`romanize`/`restore`
  byte-slicing paths (~300k cases) over adversarial Tamil + keyword + ASCII input.
- **Result:** NO overflow, NO `unsafe`, NO panic/crash. `reskin` byte-slicing is
  defensively clamped and char-aligned; `morph::localized_msg` already guards bad
  spans with `src.get(..).unwrap_or("")`; the `_2`/`_3` uniquing counter and
  string capacities are bounded.
- **F1 (fixed):** `--names-map` accepted any `NameMap.version` despite the field's
  documented purpose. `load_name_map` now rejects an unknown version with a clean
  "fail closed" error.
- **F4 (fixed):** Tamil script can be the ONLY separator between a numeric literal
  and a following keyword/identifier (`42தொகுதி`, `42கணக்கி`); reskinning to ASCII
  glued them into an unlexable lexeme (`42module`, `42kannakki`). Added a
  byte-level boundary guard (`push_guarded`) to BOTH the keyword and identifier
  arms of `reskin` — inserts one separating space exactly when a re-emitted,
  script-changing token would touch an ASCII word byte. Such input now round-trips
  token-equivalent (gains a space, not byte-identical). Not a security/overflow
  bug; a correctness edge on non-idiomatic input.
- **Docs:** caveated the "lossless round-trip" claims (spec/04, 13-tooling,
  examples/README, this log, 10-test-map) to "byte-identical for whitespace-
  separated code; token-equivalent with a separator space for number-abutting-
  Tamil". Coverage gap CLOSED: added a fourth cargo-fuzz target
  `fuzz/fuzz_targets/translate_roundtrip.rs` (keyword reskin, `--romanize-names`,
  and name-map restore must re-lex and be token-equivalent), wired into both CI
  fuzz jobs (per-PR 60s + weekly 600s) and the corpus seed. CI/Linux-only —
  libFuzzer does not build on Windows, so it is unverified locally (same as the
  other three targets); the same invariants are covered locally by the
  deterministic regression test in `tests/translate.rs`.

Tests (+2, 286 → **288**): +1 `translate` integration (boundary-guard
regression), +1 `config` integration (unknown map version rejected). Gate clean
