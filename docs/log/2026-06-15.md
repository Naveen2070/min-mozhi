# 2026-06-15 тАФ Robustness follow-up to the 2026-06-14 batch audit

A three-agent audit (plus manual code verification) of the 2026-06-14 batch
(built-ins, `fmt`, `--lang`/`morph`, LSP localization, `translate --order`,
security hardening, fuzz) found **no correctness bugs** and confirmed the

follow-up that closed the one real robustness issue, two low-severity edge
cases, and the cheap missing-test / doc-drift items the audit surfaced.

## Production fixes

- **Atomic `mimz fmt` write** (`src/main.rs` `fmt_file`). `fmt` overwrote its
  input in place with `std::fs::write`, so an interrupted / failing write could
  truncate the user's source. Now writes a sibling `<out>.<pid>.tmp` then
  `std::fs::rename`s over the target (cross-platform atomic replace; the PID
  suffix avoids concurrent-run collisions). The write still happens only after a
  successful lex + translate, so malformed input never reaches it.
- **`morph::inflect` empty-stem guard** (`src/morph.rs`). An empty `name` (e.g.
  a diagnostic whose span did not resolve, via `localized_msg`'s
  `src.get(..).unwrap_or("")`) produced a suffix-only fragment like `роР тАж`. Now
  returns empty for an empty stem.

## Mixed-flavor warning (W0001) тАФ the first non-fatal lint

After a feasibility discussion on SystemVerilog/CPU support and Tamil-script
identifiers (findings: synthesizable HDL is the same Verilog pipeline industry
uses тАФ Min-Mozhi already emits Verilog-2005, so the gap is maturity not
architecture; OOP/classes are verification-only and non-synthesizable; Tamil
identifiers already compile via the emit-time transliterator), the user asked
for a rule reflecting that **English and Tanglish share code order (SVO) and may
mix, but Tamil reads differently** тАФ so mixing Tamil keywords with the others
should be flagged. Implemented as a **non-fatal warning**, surfaced in the CLI
and the LSP.

- **`Diag` gained a `Severity`** (`Error`/`Warning`, `src/diag.rs`). `Diag::new`
  defaults to `Error` (every existing call site unchanged); `as_warning()` flips
  it. The renderer prints `error[..]` / `warning[..]` per severity; `JsonDiag`
  gained an additive `"severity"` field. The decision was a real fork тАФ the diag
  system was errors-only and `fmt --strict` warned via a raw `eprintln!` the LSP
  can't see, so a true `Diag`-level warning was needed to reach editors.
- **`morph::flavor_mix_warning(tokens)`** тАФ fires iff Tamil is present **and**
  `flavors_used` has >1 flavor (en+tanglish alone stays clean). Span points at
  the first keyword of the minority flavor. Code **W0001**; reuses
  `morph::flavors_used`/`majority_flavor`.
- **Non-fatal wiring** (`src/main.rs`): `check`/`compile`/`eval` compute the
  warning after lexing, render it alongside any errors, and exit FAILURE **only
  if some diagnostic is an error** (`Output::exit_for`) тАФ a warning-only run
  succeeds and still writes Verilog. `examples/mixed/*` now emit W0001 on every
  compile (they deliberately mix all three) тАФ exit stays 0, Verilog unchanged,
  the byte-identity test still passes.
- **LSP** (`src/lsp.rs`): `to_lsp` maps `Severity::Warning тЖТ
DiagnosticSeverity::WARNING`; `recheck` appends the per-file mix warning.
- **W-codes** are deliberately outside `ALL_CHECKER_CODES` (no forced
  error-fixture; the catalog regex matches `E`-rows only).

### Decision: fully-Tamil examples as an additive showcase, not a folder migration

- **Context:** the user wanted examples written entirely in Tamil (keywords AND
  identifiers). The tempting route тАФ making each flavor folder language-pure тАФ
  was investigated.
- **Decision:** keep the four-flavor byte-identical corpus untouched; add a
  separate `examples/tamil-pure/` showcase validated by equivalence-to-counterpart
  - its own goldens/testbenches. Add identifier romanization to `translate` as an
    **opt-in** flag only.
- **Why:** language-pure folders are a **breaking change** тАФ localized
  identifiers transliterate to different Verilog, so they cannot be byte-identical
  (violates **R9**), and they break the `mimz translate` oracle in
  `tests/translate.rs` (which asserts `translate(english/X)` matches committed
  `tamil/X` token-for-token, and `TokKind::Ident` carries the name). The rule is
  also referenced across ~46 files incl. `spec/03`, `spec/04`, architecture. The
  additive route delivers fully-Tamil code with **zero** breakage. Opt-in
  romanization keeps the lossless round-trip the default, since transliteration is
  one-way. Rejected: full pure-folder migration (too large, breaks R9 + the
  translate oracle); automatic romanization (would make the round-trip lossy by
  default).
- **Impact:** new `examples/tamil-pure/` + goldens + testbenches; `tests/`
  (`examples.rs` +2, `icarus.rs` +1 via a shared `run_self_checking` helper,
  `translate.rs` +3 integration, `src/translate.rs` +2 unit); `src/translate.rs`
  (`translate_opts`/`TranslateOpts`/`build_rename_map`); `src/emit_verilog`
  (`romanize` exposed); `src/main.rs` (`--romanize-names`). Docs: **R9** gained a
  language-pure-showcase exception (`docs/RULES.md`); `spec/04` section 2 + this
  log document the flag; `docs/code/13-tooling.md`, `examples/README.md`, and
  `docs/code/10-test-map.md` updated.

### Decision (planned, not built): per-file sidecar name-map for lossless romanized round-trips

- **Context:** `--romanize-names` is one-way, so `tamil тЖТ tanglish тЖТ tamil` with
  the flag no longer restores the exact Tamil names.
- **Decision:** a future change will emit a **per-file** `<out>.names.json`
  (bundler/ProGuard style) mapping `romanized тЖТ original Tamil` (capturing the
  `_2`/`_3` uniquing); reverse translate reads it via `--names-map` to restore
  exact names. Per-file, not central (avoids cross-file collisions).
- **Why:** the user asked for it; deferred to keep this change reviewable and to
  design the map format properly on its own. Recorded here so it is not lost.
- **Impact:** none yet тАФ next session.
