## Decisions

### Decision: `include` as a synonym alias, English column only

- **Context:** user wants both `import` and `include` to work. The
  keyword table previously enforced "one canonical spelling, no aliases".
- **Decision:** alias mechanism as DATA (per-column `*_aliases` lists in
  `keywords.toml`); `include` → `KW_IMPORT`, English column only.
  Canonical word stays `import`; future `mimz translate`/`fmt` normalize
  aliases to canonical.
- **Why:** both verbs are common enough that either should "just work"
  (beginner-first, G1). English-only because inventing Tanglish/Tamil
  synonyms before the native-speaker review would pre-empt it; flavors
  mix freely, so `include` is usable in any file anyway. The
  no-aliases rule was aimed at _spelling variants_ (dialect risk) —
  deliberate synonyms are a different thing; spec/03 now says so.
- **Breaking change:** `include` stops being a legal identifier (pre-1.0,
  acceptable; did-you-mean economy unaffected).
- **Impact:** spec/02 + spec/03 → v0.2.1; keywords.toml schema; lexer
  table loader; docs/code/02.

### Decision: width semantics settled with the width pass (spec v0.2.2)

- **Context:** implementing E04xx forced answers the spec left open.
  The first three were put to the user explicitly; the rest follow the
  spec's own explicit-over-implicit philosophy.
- **Decisions (user-chosen):**
  1. **`bit` ≡ `bits[1]`** — one type, two spellings; `data[i]` connects
     to a `bit` port without ceremony.
  2. **Lossless `+`/`-` accept unequal widths** (result `max + 1`;
     `*` = sum) — growth can never lose information, so requiring
     equality would add noise, not safety.
  3. **`extend`/`trunc` allow the same-width no-op** — the strict
     version breaks parameterized code at boundary bindings
     (`extend(din, WIDTH)` at `WIDTH = 1`).
- **Decisions (implementer, spec-philosophy-driven):** `trunc` keeps the
  LOW N bits; shift amounts are constants or unsigned signals, never
  `signed`; `match` rejects `signed` scrutinees (patterns cannot say
  `-1`); concat parts must be `bits` (cast `signed` visibly); slicing
  `signed` yields `bits`; param defaults may reference earlier params
  (left to right); a compile-time operand of a LOSSLESS op that doesn't
  fit the other side takes its own minimal width instead (growth is
  safe); instantiation-config worklist capped at 1000.
- **Impact:** spec/02 v0.2.2; `src/checker/widths.rs`;
  docs/code/11-checker.md catalog; `examples/*/shift_register.mimz`.

### Decision: language drivers = one Rust core + thin wrappers, parked in Phase 4

- **Context:** idea raised — expose Min-Mozhi inside TS/JS, Python,
  Kotlin, Go, Java, Rust pipelines ("drivers") so devs in those
  ecosystems can write and run Min-Mozhi from their own build tools.
- **Decision:** the idea is GOOD but its shape is fixed now to prevent
  the expensive version later: there will only ever be **one compiler
  core** (this Rust crate) producing two distributable artifacts — the
  static CLI binary and a WASM build. Every language "driver" is a thin
  wrapper (~100 lines) around those, the esbuild/protoc model — never a
  reimplementation, never six codebases. Execution order:
  (1) WASM browser playground, (2) npm wrapper, (3) PyPI wrapper,
  (4) others only on demonstrated demand. All of it lives in Phase 4
  (`docs/plan/phase-4-ecosystem.md`, new "Ecosystem drivers" section);
  nothing is built before Phase 1.5, because "run Min-Mozhi in your
  pipeline" needs the simulator to mean anything.
- **Why parked:** solo dev, ~8–10 h/wk, Phase 1 due 31 Dec 2026 — six
  release pipelines is a permanent tax that would come out of compiler
  time; and the constitution's persona (TN polytechnic student) is
  served most by the no-install browser playground, not by Java
  bindings.
- **What Phase 1 carries so the door stays open:** keep the core
  embeddable — a lib/bin split (move `project.rs` printing into the CLI
  shell) and a `--json` diagnostics flag, folded into the already-planned
  lexer/parser E-code retrofit.

### Decision: every example exists in all four flavors, identically named

- **Context:** examples were flat and almost entirely English; the
  trilingual thesis was only spot-checked (one Tanglish counter).
- **Decision:** four flavor folders with the same base examples,
  identical filenames and identifiers; `mixed/` deliberately mixes
  flavors per file. Tests enforce the structure: everything compiles,
  and per-example Verilog is byte-identical across the four folders.
- **Why:** turns "one AST, three skins" from a claim into a CI-asserted
  property over the whole example corpus; gives Tamil/Tanglish learners
  first-class examples instead of one token sample.
- **Impact:** examples/ layout; tests/examples.rs `BASE_EXAMPLES` const
  is the ledger; docs/code/10 house rule: new examples land in all four
  folders + the const, same commit.
