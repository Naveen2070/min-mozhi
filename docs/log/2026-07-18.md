## BUG-11 fixed — bigger than originally scoped

- Stage 1 started with BUG-11 (simulator vs. synthesis mismatch on `<<`).
  The review's own prescribed fix ("truncate the result to `l.width`
  unconditionally") was tried first and **empirically disproven** against
  `iverilog` before landing: `din << 2` (`din: bits[4]`) assigned to an
  8-bit target computes **28** in real Verilog (context-extends `din`
  before shifting), not **12** (truncate-then-extend, what the review's own
  fix would have produced) — caught immediately by the pre-existing Icarus
  differential test for `english/shift.mimz`'s `var_shift`, which the
  review's fix broke.
- Real rule (ground-truthed, not assumed): Verilog's `<<`/`>>` are
  **context-determined** on their left operand — the operand widens to the
  enclosing assignment/`extend` target BEFORE the shift, not after. Neither
  "grow by the shift amount" (BUG-6's original fix, which is what caused
  BUG-11) nor "always truncate to `l.width`" (this review's fix) match
  Verilog in general; only threading the real context width through does.
- Implemented `eval_ctx`/`binary_ctx` (`crates/mimz-sim/src/sim/value.rs`)
  — an `expected_width: Option<u32>` parameter used only by `Shl`/`Shr`
  (every other operator's width rule is deliberately unchanged, scoped to
  this bug). Wired into every call site with a real known target width:
  combinational driver resolution (`comb.rs`, `kernel.rs`), register/
  default/memory-cell writes, `Builtin::Extend`'s argument (the exact site
  that exposed the `din << 2` case), and `fn`-body `let`s (reusing the
  checker's existing `inferred_width` cell). `if`/`match` propagate the
  same context into every branch. Both the review's own `(a<<2)>>2` repro
  (63, not 255) and the `din<<2` case now match Icarus. Old test
  `shl_does_not_truncate_to_left_operand_width` (which asserted the BUG-6
  overcorrection this bug's own root cause) replaced with 3 tests covering
  self-determined, context-widened, and chained-shift-shares-one-context
  behavior. `docs/audit/bugs.md`'s BUG-11 marked FIXED with full detail;
  `security.md`'s SEC-9 marked FIXED (SEC-8 — the reachability class itself
  — stays OPEN, this closed one instance of it, not the class).

## BUG-10 returns — diagnostic instead of invalid Verilog

- Stage 1 continued with BUG-10's still-open return half. First attempt
  rejected bundle-typed `fn` returns in the checker
  (`check_func_body_widths`) — wrong, and reverted before landing: feature
  2.9's structural interface matching already has full, deliberately-built,
  tested checker-level support for bundle-typed `fn` returns
  (`check_return_ty`'s `BundleShapeMatch` handling, E0910/E0804 etc.).
  Rejecting there broke several existing, legitimate tests
  (`structurally_compatible_fn_return_checks_clean` and others) — would
  have fixed nothing, just destroyed working validation.
- Real gap is narrower: `width_subst`
  (`crates/mimz-core/src/emit_verilog/module.rs`) is reached **exclusively**
  via `render_fn_decl`'s `let ret_w = self.width(&decl.ret)` — every other
  caller (module ports/wires, `fn` params) flattens a bundle to per-field
  signals before ever calling `width()`/`width_subst()`. Its `Type::Bundle`
  arm silently returned `String::new()` (parametric form); its
  `Type::Named` arm's bundle case fell through to a misleading "not a
  declared enum" message (bare form). Both now report a real diagnostic.
  `mimz check` still accepts a bundle-typed `fn` return cleanly (the
  checker's own view is correct and unchanged); `mimz compile` now rejects
  it — the same check-vs-compile split the bare form already had, now
  consistent for both forms.
- Confirmed via `docs/plan/phase-2-ir-synthesis.md` that the real fix
  (call-site inlining, since a Verilog `function` can only return one
  value) is already tracked in the language-features backlog — not
  duplicated.
- 2 new tests plus one repurposed (the old
  `structurally_matched_fn_return_emits_same_as_nominal_match`, renamed to
  `..._is_a_diagnostic_same_as_nominal_match`) — its old dummy-`W`-param
  workaround for sidestepping the hard-error path no longer works now that
  BOTH forms are rejected; repurposed to the still-meaningful invariant it
  was really pinning: nominal and structurally-matched bundle returns get
  the identical diagnostic.
