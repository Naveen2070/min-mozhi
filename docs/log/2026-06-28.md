### Decision block (R3)

**What.** Combinational user-defined functions: `fn f(params) -> T { [let x = e ‚Ä¶]
bodyExpr }`, called as `f(a, b)` in any expression. File-level only (not inside
modules or other functions). Pure combinational ‚Äî no clocks, no registers, no
module instantiation.

**Why file-level only (D3).** Keeping functions at file scope mirrors Verilog's
`function automatic`, simplifies scope resolution (functions can't capture module
state ‚Äî they see only their parameters and `let` locals), and avoids the question
of whether a function declared inside a module could reference that module's ports
(it cannot ‚Äî the only inputs are explicit parameters). A function needed by two
modules sits at file level and the emitter injects it into each using module.

**Why monomorphic (D5).** Functions are not generic/parametric ‚Äî a call `f(x)` must
resolve to exactly one `FuncDecl` at compile time. No instantiation, no type
parameters. This keeps the checker and emitter simple: one width-check per
declaration, one Verilog body per function.

**Why recursion is banned (E0805).** Combinational circuits are acyclic by
definition. A recursive `fn` call would create a combinational loop, which the
existing DAG checker (E0504) would catch at module level anyway ‚Äî banning it at the
function level gives a clearer, earlier error (E0805 "recursive fn" is more
informative than E0504 "combinational cycle through a function").

**Why `let` locals are typed as `integer` in emitted Verilog.** `LocalLet` carries
no `ty` field (the type is inferred from the right-hand expression). Emitting
`integer` (32-bit signed Verilog) is a deliberate compromise: it covers most
intermediate values in practice and avoids a partial-width-inference pass. Width
inference for locals is a follow-up (requires an AST change to add a `ty` field
and a new checker sub-pass). Annotated with a `// ponytail:` comment in the
emitter.

**Why `function automatic` (not `function`).** Verilog-2005 `function` uses static
local variables ‚Äî calling it twice with different args can alias. `function
automatic` uses stack-allocated locals per call, matching the expected purely
functional behavior. Static Verilog functions would be a silent semantic bug.

**Keyword.** `fn` (English), `function` (alias, familiar to web/systems developers),
`saarbu` / `ýÆöýÆæýÆ∞ýØçýÆ™ýØÅ` (the Tamil mathematical term for "function") ‚Äî PROVISIONAL
Tanglish/Tamil, pending native-speaker review (R9/R11).

**E-codes.** E0801 (duplicate fn name), E0802 (fn name shadows builtin),
E0803 (arity mismatch), E0804 (return-type width mismatch), E0805 (recursion);
E1110 reused for unknown fn name at call site.
All reuse the existing E0xxx/E1xxx namespace (no new code block needed).

### Decision block (D1‚ÄìD6)

**D1 ‚Äî Tag-only variants omit parens.** `A` (not `A()`) for variants without
payload. Empty parens `A()` are a parse error. Reason: `A()` could be
misread as a zero-width field, and the physical layout would allocate 0 bits
for payload ‚Äî ambiguous with a tag-only variant.

**D2 ‚Äî Positional bindings (not named).** Match patterns use
`Enum.Variant(b1, b2)` not `Enum.Variant(name: b1, ...)`. Reason: variant
fields are already named in the declaration; repeating names at the match
site would be redundant. The checker enforces arity (E0806) and the binding
order matches declaration order.

**D3 ‚Äî Packed MSB-first layout.** Payload fields are packed MSB-first inside
the `max_payload_w` region. Field `i` occupies bits
`[max_payload_w - 1 - offset_i : max_payload_w - 1 - offset_i - width_i]`
where `offset_i` is the cumulative width of preceding fields. Tag occupies
the MSB bits `[total_w - 1 : total_w - tag_w]`. Reason: MSB-first means the
tag is always in the highest bits regardless of payload width ‚Äî simpler for
manual inspection and waveform debugging.

**D4 ‚Äî `inferred_total_width` is a `OnceCell`.** Set once by the checker.
The emitter and sim read-only. If the checker hasn't run (bare sim tests),
tag-only enums fall back to `clog2(n_variants)` which equals the correct
width. Tagged enums without checker hit a `debug_assert!` ‚Äî a safety net
that catches out-of-order pipeline execution in debug builds.

**D5 ‚Äî `Pattern::Variant` lowered to `IntMask` in sim.** After elaboration,
a `Packet.Read(a)` pattern becomes `IntMask { value: tag_val << max_payload_w,
mask: tag_mask << max_payload_w }`. The raw `Pattern::Variant` never reaches
the `pattern_matches` evaluator ‚Äî that path is `unreachable!()`. Reason:
avoids duplicating payload-width logic in the evaluator; the elaborator
handles it once.

**D6 ‚Äî Nested enums as payload deferred.** A payload field of enum type is
E0807 ("non-concrete type"). Nested tagged enums require recursive width
computation and are left for follow-up. Only `bit`, `bits[N]`, `signed[N]`
are valid payload types.
