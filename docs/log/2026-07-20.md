# 2026-07-20

## Stage 4 — BUG-23 fixed (wrap-operand hoist everywhere), BUG-24 found/misdiagnosed/corrected/fixed, two landing-time regressions caught by a full-workspace gate

Closed out BUG-23 (filed yesterday during Phase A1b's own closing-the-loop
fuzzer run) via `docs/superpowers/plans/2026-07-19-bug23-wrap-hoist.local.md`,
5 tasks, subagent-driven. Root cause: a wrapping operator (`+%`/`-%`/`*%`)
nested as a direct operand of a DIFFERENT, sibling context-determined
operator (`+`/`-`/`*`/`&`/`|`/`^`) loses its own-width truncation — not
one of the four Verilog self-determined positions Phase A1b's hoist
mechanism actually checks (concat member, replication part, comparison
operand, `$signed`/`$unsigned` argument), since "operand of another
binary operator" isn't itself a self-determined position in the Verilog
LRM sense at all. `infer_kind`/`verilog_self_determined_kind` (A1b's
existing mismatch-detection machinery) can never catch this shape, because
neither function ever gets consulted at that position in the first place
— the bug isn't a wrong answer to a question A1b asks, it's a question
A1b never asks there.

**The fix**: `Emitter::hoist_width_effect_operand`
(`crates/mimz-core/src/emit_verilog/expr.rs`), extracted from the
pre-existing hoist pattern already used by `Builtin::Extend`'s own render
arm (Task 1, pure refactor — `Builtin::Extend` had already solved this
exact problem for its own one call site, just never generalized). Wired
into the shared binary-operator arm's non-comparison branch (Task 2, the
primary gap this bug named) and then into every remaining recursive-
descent operand position `expr_subst` walks into — unary, `if`/`match`
branches, concat/replicate members, index base, function-call/enum-
construct arguments, and the remaining builtins (Task 3). Any lossless-
or-wrap-family operand sitting at any of these positions now gets hoisted
into a fresh, definite-width wire unconditionally — regardless of whether
it also happens to land in one of A1b's four checked positions. Only the
true top-level assignment RHS stays exempt.

Two sub-fixes landed along the way, both found while implementing the
wiring, not part of the plan's original text:

- **Signedness gap.** `hoist_slice_base_if_needed` always declared its
  hoisted wire plain unsigned. A signed wrap operand hoisted through it
  would get zero-extended instead of sign-extended once the surrounding
  operator ran at its own wider context (Verilog LRM 5.5.1 — any unsigned
  operand makes the whole expression unsigned), silently changing the
  value. Fixed by adding a `signed: bool` parameter, mirroring
  `hoist_if_needed`'s existing correct pattern. New regression test:
  `bug_23_signed_wrap_operand_hoist_preserves_sign_extension`.
- **Double-hoist.** Four call sites (`Concat`/`Replicate`/`SignedCast`/
  `UnsignedCast`) run BOTH the new unconditional hoist and the existing
  self-determined-mismatch hoist (`hoist_if_needed`) on the same operand.
  A lossless operand matched by both got hoisted twice — a wire, then a
  same-width alias wire pointing at it. Fixed by giving `hoist_if_needed`
  the same `is_plain_identifier` early-return guard `hoist_slice_base_if_needed`
  already had (a rendered text that's already a bare identifier needs no
  further hoist, its declared `Kind` already trivially matches). New
  regression test: `bug_19_lossless_sub_in_a_concat_hoists_exactly_one_wire`
  (a wrap operand alone can't prove this — `matched_result` and
  `verilog_self_determined_kind` are provably always equal for wrap, so
  `hoist_if_needed` is a guaranteed no-op there regardless of the guard; a
  LOSSLESS operand was needed to actually exercise the double-hoist path).

Task 4's default-N differential-fuzz pass confirmed BUG-23 itself fixed
(4/4) — the exact tests that originally caught it.

## BUG-24 found (Task 4's deep-N pass), misdiagnosed once, then correctly root-caused and fixed

Task 4's `MIMZ_DIFF_FUZZ_N=500` deep-confidence pass (the completeness
proof this plan's own design called for, given how many call sites Task 3
touched) found a DIFFERENT failure at seed `12648537` — a width-growing
operator (`p1 * p1`) losing its growth as the left operand of a shift
(`<<`). The **first-filed diagnosis was wrong**: it claimed the root
cause was a missing "fifth self-determined position" (a shift's left
operand) that `emit_verilog/self_determined.rs` needed to be taught,
mirroring BUG-19's original fix shape. Hands-on empirical verification
against real Icarus (`p1 << 3` evaluated at its own natural width vs. a
wider context genuinely gives different values) found a simpler, already-
familiar root cause instead: this is the SAME bug class as BUG-23, for
two operators BUG-23's own fix didn't cover. `is_width_effect_binop`
(`expr.rs`) wrongly excluded `Shl`/`Shr` on the assumption — stated in its
own doc comment — that every operator besides lossless/wrap "gives the
SAME value no matter what width Verilog happens to (re)compute it at."
False for shift: a shift's RESULT width is self-determined (fixed to the
left operand's own width, unaffected by context — this much of the
original diagnosis was right), but the shift's LEFT OPERAND is itself
context-determined — real Verilog widens it to the ambient context BEFORE
shifting. `mimz-sim`'s own `eval_ctx` already models this correctly
(ground-truthed against Icarus since BUG-11); the emitter's hoist
mechanism had just never extended the same fact to `is_width_effect_binop`.
The wrong diagnosis was withdrawn in-place in the entry itself (not
hidden) once the real cause was confirmed — an honest correction, useful
to keep visible: the first theory would have led to touching the wrong
file (`self_determined.rs`) for no reason.

**Fix**: added `BinOp::Shl | BinOp::Shr` to `is_width_effect_binop`'s
match arm, corrected its doc comment, and let the existing Task 1-3
hoisting machinery cover shift the same way it already covers lossless/
wrap — no other code changes needed at first. New regression test:
`bug_24_shl_under_sibling_add_matches_icarus`.

## Two further regressions — found by Task 5's own first attempt, both fixed before landing

Task 5's job is full verification + docs. Its own FIRST full-workspace
run (`cargo test --workspace --all-targets --no-fail-fast`) — something
none of Tasks 1-4 had actually done, each having scoped its own
verification to `self_determined_regression`/`differential_fuzz`/a
targeted emitter suite — surfaced two more regressions. This is exactly
the kind of thing a full-workspace gate exists to catch, and exactly why
it's worth recording as a process point, not just a bug list: individual-
task verification, scoped to the suites each task's own plan named, had
missed both.

1. **Generic-width decls fallback bug.** `resolved_kind`
   (`emit_verilog/module.rs`) silently defaulted an unresolved generic-
   parameter width (e.g. a module's own `WIDTH: int = 8` generic feeding a
   `bits[WIDTH]` port) to 1 bit. This had presumably been latent for a
   while, but Tasks 2-4's new hoist call sites (plus BUG-24's `Shl`/`Shr`
   addition) started actually REACHING that fallback and silently
   truncating real hardware — confirmed against `alu.mimz` and
   `shift_register.mimz` via Icarus, both producing visibly wrong
   waveforms. Fixed by changing `resolved_kind`'s return type to
   `Option<Kind>` and having `build_decls` skip inserting a decl entirely
   when a width can't be resolved, instead of substituting a wrong
   value — `kind_is_inferrable`'s existing check then naturally refuses
   to hoist at a position with no resolvable width, rather than hoisting
   to a wrong 1-bit width.

2. **Shift-hoist applied too broadly.** BUG-24's own fix (`Shl`/`Shr`
   added to `is_width_effect_binop`) initially hoisted a shift at EVERY
   call site the existing machinery touches — but the reference simulator
   only self-determines a shift's left operand at SOME positions
   (`Concat`/`Replicate` members, a plain `Index` base, most builtin
   arguments); at `Builtin::Extend`'s own argument, `IfExpr`/`Match`
   branches, and a shift's LHS when the OUTER operator is itself a shift,
   the simulator threads real ambient context through instead, and
   hoisting there computed the WRONG value. This regressed
   `examples/english/shift.mimz` — BUG-6's own historical guard, from
   long before this session. Fixed by splitting a new `is_shift_binop`
   out of `is_width_effect_binop` (the lossless/wrap family stays
   unconditionally hoistable, unchanged) and adding an `allow_shift: bool`
   parameter to `hoist_width_effect_operand`, individually classified per
   call site against the simulator's actual source (`false` at the 4
   unsafe positions, `true` everywhere else — full per-site rationale now
   lives in `is_shift_binop`'s own doc comment and inline comments at each
   call site). Two new regression tests:
   `bug_24_regression_shift_in_if_branch_stays_unhoisted` and
   `bug_24_regression_nested_shift_lhs_of_shift_stays_unhoisted`.

Both fixes were reviewed and approved (no Critical/Important findings),
including a specific check that running both fixes concurrently in the
same non-isolated working tree (one touching `module.rs`, one touching
`expr.rs`) left nothing corrupted or half-migrated.
